# AGENTS.md

## 1. 项目身份与当前状态

本仓库是 **AI 产品工程框架（AI Product Engineering Framework）**，不是 Prompt 合集、代码模板堆积、Skill 收藏库或某个厂商的专属配置仓库。

项目目标是建立一套开放、跨平台、可验证的产品工程方法，使 Claude Code、Codex、Kimi、GLM 等 AI Agent 能够在人的目标与判断下，受控参与价值验证、产品定义、体验设计、高保真预览、工程实现、质量验证、用户验收、发布反馈和持续演进。

```text
当前稳定版本：v0.1.1
目标开发版本：v0.2.0
当前开发里程碑：A / Context 可执行化
```

- Framework 自应用 Context 已通过并由项目维护者确认；
- Context 规范和候选模板已建立，但尚未经过真实业务参考工程验证；
- 候选资产不得描述为稳定能力。

## 3. 不可破坏的核心模型

框架由三个平面组成，禁止将它们重新混成一条扁平的“十几层流水线”。

### 3.1 产品价值生命周期

根据 `DEC-005`，统一为十个阶段：

1. 战略与价值验证；
2. 产品定义；
3. 用户体验设计；
4. 高保真原型预览与确认；
5. 工程规格设计；
6. 受控任务执行；
7. 质量与安全验证；
8. 模拟用户验收；
9. 发布交付；
10. 运行反馈与持续迭代。

模拟用户验收和发布交付不得再次合并：验收证明产品可用，发布批准版本进入目标环境。

### 3.2 AI 工程基础设施

- Context Engineering：让 AI 获得正确、必要、分层和可追溯的项目事实；
- Harness Engineering：让 AI 在阶段、权限、边界、契约和门禁内执行；
- Skill Engineering：把真实验证过的方法封装成可重复能力；
- Agent Engineering：定义承担目标的角色、输入输出契约和协作方式；
- Loop Engineering：根据结果、数据和反馈进行观察、评估、纠偏、停止和沉淀。

五类能力贯穿整个生命周期，不是产品阶段。

### 3.3 治理与组织记忆

- 设计决策、版本和事实状态；
- 安全、隐私、权限和合规；
- 质量、指标和可观测性；
- 成本、变更、发布和回滚；
- 参考工程对框架的验证反馈。

## 4. 人与 AI 的责任边界

### 人必须负责

- 目标、价值与优先级判断；
- 产品范围和关键取舍；
- 高保真体验确认；
- 敏感数据、权限和高风险变更审批；
- 验收豁免、发布和回滚决定；
- 最终产品、工程和社会责任；
- 开源许可证等法律与治理选择。

### AI 可以负责

- 检索、分析、归纳和方案比较；
- 文档、原型、代码、配置、测试和报告生成；
- 任务拆解和受控工具执行；
- 自动检查、测试、问题定位和候选修复；
- 数据、反馈、失败和成本分析；
- 将验证过的经验整理为候选 Context、模板、门禁或 Skill。

AI 不得用“已完成”“应该没问题”“高置信度”等主观判断代替验证证据和人工批准。

## 5. 工作顺序

任何实质性新增或修改均遵循：

```text
定位问题
→ 查阅宪法、全局模型、版本规范和设计决策
→ 确认唯一所属模块和事实源
→ 明确目标、输入、输出、边界、风险与验证
→ 建立或更新任务 Context Pack
→ 修改
→ 自检和运行验证
→ 更新关联专题文档、导航和版本状态
→ 记录需要长期保留的决策或经验
```

禁止：

- 未查阅关联文档就新增平行概念；
- 建立与现有权威目录同义的平行目录；
- 为追求完整感而批量建设没有参考工程验证的 Skill、脚本或平台；
- 将工具、Agent、Skill、流程、生命周期阶段和组织角色混用；
- 只更新 README，不更新作为事实来源的专题文档；
- 让关键结论只存在于对话、Issue、图片或提交信息中；
- 用局部实现静默改变生命周期、核心原则或责任边界；
- 将内部开发里程碑误写成发布版本；
- 由 Agent 自行选择或更换开源许可证。

## 6. Context 与记忆规则

仓库和受控知识资产是项目长期记忆的事实来源，模型会话不是。

信息优先级：

1. 当前有效且未被替代的设计决策；
2. 当前版本宪法、专题文档和正式契约；
3. Schema、高保真确认和验证记录；
4. 仓库级 Agent 指令；
5. 已批准的项目、阶段和任务 Context Pack；
6. 临时对话和模型推断。

发生冲突时不得静默选择，必须指出：

- 冲突的事实源；
- 各自版本、状态和适用范围；
- 建议保留、替代或归档的结论；
- 是否阻塞当前任务；
- 是否需要新的设计决策。

以下变化必须记录设计决策：

- 框架核心模型或宪法变化；
- 模块边界和单一事实源变化；
- 关键术语含义变化；
- 生命周期阶段增删、拆分或合并；
- 核心门禁、权限或责任归属变化；
- 平台绑定或重大技术选择；
- 自动化等级和高风险人工介入点变化；
- 发布版本和开发里程碑语义变化。

## 7. 版本、状态和成熟度规则

### 7.1 发布与开发

- 发布版本：`v主版本.次版本.修订版本`；
- 当前稳定版本：v0.1.1；
- 当前目标版本：v0.2.0；
- `A/B/C` 只表示 v0.2.0 内部里程碑；
- Review 和一致性修订增加修订版本；
- 新增一组可验证能力增加次版本。

### 7.2 Context 版本字段

必须区分：

- `context_pack_version`；
- `project_context_pack_version`；
- `stage_context_pack_version`；
- `task_context_pack_version`；
- `stable_release`；
- `target_release`；
- `source_version`；
- `source_commit`。

禁止使用含义不清的通用 `version` 表达多个概念。

### 7.3 机器状态

YAML 和自动门禁使用统一英文机器值，文档正文可以显示中文：

- 事实源：`draft`、`active`、`superseded`、`archived`、`rejected`；
- 阶段：`not_started`、`preparing`、`active`、`in_review`、`approved`、`blocked`、`stopped`；
- 任务：`draft`、`pending_confirmation`、`ready`、`in_progress`、`verifying`、`pending_human_approval`、`completed`、`blocked`、`cancelled`；
- 成熟度：`candidate`、`single_project_validated`、`cross_project_validated`、`stable`、`deprecated`。

## 8. 文档与命名规范

- `README.md`、`AGENTS.md`、`CHANGELOG.md`、`CONTRIBUTING.md` 等通用规范文件保留英文文件名；
- 其他目录和文档优先使用清晰、稳定的中文名称；
- 正文以中文为主，必要行业术语保留英文并首次解释；
- 不为了“国际化”复制内容相同的英文目录；
- 不使用 `Demo`，统一使用“样例”“参考工程”或“验证工程”；
- 一个概念只能有一个当前权威文档，其他文件通过链接引用；
- 删除或替代文档前必须确认内部链接和设计决策是否同步；
- 提交信息使用中文，例如：`新增：...`、`完善：...`、`修复：...`、`重构：...`、`发布：...`。

## 9. 图示规范

复杂概念优先采用“真实图片资产 + Mermaid + 文字解释”三层表达。

- 真实图片使用仓库内 SVG/PNG，不依赖易失效外链；
- ChatGPT 生成图片用于快速理解和传播，必须标明正式定义的权威来源；
- Mermaid 用于可维护的结构、流程、状态和关系图；
- 每张图只表达一个核心问题；
- 图下必须解释图要证明什么；
- 图片路径、Markdown 链接和 Mermaid 必须经过检查；
- 高保真产品预览是生命周期中的正式门禁，不等同于框架宣传图。

## 11. 交付要求

1. 修改了哪些文件；
2. 解决了什么问题；
3. 是否改变核心模型或事实源；
4. 稳定版本、目标版本和成熟度是否变化；
5. 验证方式和结果；
6. 尚未解决的风险；

8. 下一步应进入哪个模块，而不是泛化为“继续完善”。
