# AGENTS.md

## 1. 项目身份

本仓库是 **AI 产品工程框架（AI Product Engineering Framework）**，不是 Prompt 合集、代码模板堆积、Skill 收藏库或某个厂商的专属配置仓库。

项目由 zhidao-studio 专有维护，目标是建立一套跨平台、可验证的产品工程方法，使 AI Agent 能够在人的目标与判断下，受控参与价值验证、产品定义、体验设计、高保真预览、工程实现、质量验证、用户验收、发布反馈和持续演进。

当前稳定版本、目标版本、工作段、参考工程状态、阻塞和下一步，统一以 [Framework 项目 Context](12_框架项目Context/README.md) 为唯一入口。AGENTS 不维护这些易漂移的动态事实。

## 2. 修改前阅读顺序

任何实质性修改前，按以下顺序读取：

1. 根目录 `LICENSE`；
2. 当前有效且未被替代的设计决策；
3. `01_框架定义/` 中愿景、适用场景、核心原则、边界和术语规范；
4. `02_全局模型/AI产品工程全局框架.md`；
5. `10_版本演进/版本管理规范.md`；
6. `12_框架项目Context/README.md` 和当前阶段 Context；
7. 当前任务所属专题文档；
8. 当前任务 Context Pack；
9. 本文件。

README 用于快速理解和导航，不替代 LICENSE、设计决策、专题文档、项目 Context 和正式约定。

## 3. 不可破坏的核心模型

### 3.1 三个平面

框架由三个相互配合的平面组成，禁止重新混成一条扁平流水线：

1. **产品价值生命周期**：规定从价值验证到持续迭代的主流程；
2. **AI 工程基础设施**：Context、Harness、Skills、Agents、Loop；
3. **治理与组织记忆**：决策、版本、安全、权限、质量、成本、变更和参考工程反馈。

### 3.2 十阶段产品价值生命周期

根据 `DEC-005`，统一为：

1. 战略与价值验证；
2. 产品定义；
3. 用户体验设计；
4. 高保真原型预览与确认；
5. 工程规格设计；
6. 受控任务执行；
7. 质量与安全验证；
8. 模拟用户验收；
9. 发布交付；
10. 运行反馈与持续迭代。

模拟用户验收与发布交付必须分离：验收证明产品可用，发布批准版本进入目标环境。

### 3.3 五大工程基础设施

- **Context Engineering**：让 AI 获得正确、必要、分层和可追溯的事实；
- **Harness Engineering**：通过阶段、权限、边界、约定和检查关卡限制执行；
- **Skill Engineering**：把经过真实验证的方法封装成可重复能力；
- **Agent Engineering**：定义角色责任、输入输出和协作方式；
- **Loop Engineering**：根据结果、数据和反馈进行观察、纠偏、停止和沉淀。

五类基础设施贯穿生命周期，不是产品阶段。

## 4. 人与 AI 的责任边界

### 人必须负责

- 目标、价值与优先级判断；
- 产品范围和关键取舍；
- 高保真体验确认；
- 敏感数据、权限和高风险变更审批；
- 验收豁免、发布和回滚决定；
- 最终产品、工程、法律和社会责任。

### AI 可以负责

- 检索、分析、归纳和方案比较；
- 文档、原型、代码、配置、测试和报告生成；
- 任务拆解和受控工具执行；
- 自动检查、测试、问题定位和候选修复；
- 数据、反馈、失败和成本分析；
- 将验证过的经验整理为候选 Context、模板、检查关卡或 Skill。

AI 不得用“已完成”“应该没问题”或主观置信度代替验证证据和人工批准。

## 5. Context 与事实源规则

仓库和受控知识资产是长期记忆，模型会话不是。

事实优先级：

1. `LICENSE` 和有效法律治理决策；
2. 当前有效设计决策；
3. 当前宪法、专题文档和正式约定；
4. 当前项目 Context、阶段 Context、Schema、高保真确认和验证记录；
5. 仓库级 Agent 指令；
6. 已批准任务 Context Pack；
7. 历史任务、验证和发布报告；
8. 临时对话和模型推断。

发生冲突时必须指出冲突来源、版本、适用范围、建议处理、是否阻塞和是否需要新设计决策，不得静默选择。

历史文件允许保留当时术语和判断，但必须标明：

- 本文件为历史快照；
- 当前状态以项目 Context 为准；
- 被哪个后续任务、验证或版本替代。

## 6. 工作顺序

```text
定位问题
→ 读取法律、宪法、版本、设计决策和当前Context
→ 确认唯一所属模块和事实源
→ 明确目标、输入、输出、边界、风险和验证
→ 建立或更新任务Context Pack
→ 修改
→ 静态检查和能够执行的运行验证
→ 更新专题文档、导航、Context和版本状态
→ 记录长期决策或经验
```

禁止：

- 未读取事实源就新增平行概念；
- 建立同义平行目录或重复事实源；
- 批量建设未经参考工程验证的 Skill、脚本或平台；
- 混用工具、Agent、Skill、流程、生命周期和组织角色；
- 只更新 README，不更新权威专题文档；
- 让关键结论只存在于对话、图片、Issue 或提交信息；
- 用局部实现静默改变核心模型和责任边界；
- 将内部里程碑误写成稳定发布版本；
- 将历史报告误当成当前运行状态；
- 把检查已定义写成检查已执行通过；
- 把候选资产写成稳定或 `single_project_validated`；
- 提前启动未满足进入条件的后续里程碑；
- 由 Agent 自行选择、更换或放宽专有许可证；
- 将专有内容复制到未授权系统或数据集。

## 7. 版本与状态规则

### 7.1 发布版本

- 发布版本使用 `v主版本.次版本.修订版本`；
- 当前稳定版本只从根目录 `VERSION` 读取；
- 目标版本、当前里程碑和工作段只从 Framework 项目 Context 与 Roadmap 读取；
- `A/B/C` 是目标次版本内部里程碑；
- `A1/A2` 是里程碑内部工作段，不进入 `VERSION`；
- 治理、事实源和参考工程同步通常增加修订版本；
- 新增完整且经过真实验证的能力族才考虑增加次版本。

### 7.2 Context 版本字段

必须区分：

- `context_pack_version`；
- `project_context_pack_version`；
- `stage_context_pack_version`；
- `task_context_pack_version`；
- `stable_release`；
- `target_release`；
- `source_commit`；
- `baseline_commit`。

禁止使用含义不清的通用 `version` 同时表达多个概念。

### 7.3 状态与成熟度

机器状态使用稳定英文值。常用状态包括：

- 事实源：`draft`、`active`、`superseded`、`archived`、`rejected`；
- 阶段：`not_started`、`preparing`、`active`、`in_review`、`approved`、`blocked`、`stopped`；
- 任务：`draft`、`pending_confirmation`、`ready`、`in_progress`、`verifying`、`pending_human_approval`、`completed`、`blocked`、`cancelled`；
- 成熟度：`candidate`、`single_project_validated`、`cross_project_validated`、`stable`、`deprecated`。

版本号不自动提升资产成熟度。

AGENTS 不记录仓库可见性等动态平台状态，也不将其作为工程任务阻塞或发布条件。

## 9. 文档、命名与图示

- 中文正文使用“约定”和“检查关卡”；技术名称和机器字段按需保留；
- 通用文件名保留 `README.md`、`AGENTS.md`、`CHANGELOG.md`、`LICENSE` 等英文形式；
- 其他目录和文档优先使用清晰稳定的中文名称；
- 不使用 `Demo`，使用“样例”“参考工程”或“验证工程”；
- 一个概念只有一个当前权威文档，其他文件通过链接引用；
- 删除或替代文档前检查内部链接和设计决策；
- 复杂概念优先采用仓库图片、Mermaid 和文字解释；
- 高保真产品预览是正式检查关卡，不是宣传图；
- 提交信息使用中文并准确描述修改范围。

## 10. 设计决策要求

以下变化必须建立设计决策：

- 框架核心模型或宪法变化；
- 模块边界和单一事实源变化；
- 关键术语变化；
- 生命周期阶段增删、拆分或合并；
- 核心检查关卡、权限和责任变化；
- 重大平台或技术选择；
- 自动化等级和高风险人工介入点变化；
- 发布版本和开发里程碑语义变化；
- 专有许可和外部授权边界变化。

普通参考工程修复与事实同步不需要新增框架核心设计决策，除非它改变上述长期规则。

## 11. 完成要求

每次任务必须说明：

1. 修改文件；
2. 解决的问题；
3. 是否改变核心模型或事实源；
4. 稳定版本、目标版本、成熟度和许可边界是否变化；
5. 实际执行的验证和结果；
6. 未执行或失败的验证；
7. 尚未解决的风险；
8. 是否新增、替代或归档设计决策；
9. 下一步进入哪个明确任务或模块。
