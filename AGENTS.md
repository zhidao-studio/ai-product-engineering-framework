## 3. 不可破坏的核心模型

框架由三个平面组成，禁止将它们重新混成一条扁平的“十几层流水线”。

### 3.1 产品价值生命周期

根据 `DEC-005`，统一为十个阶段：

1. 战略与价值验证；
2. 产品定义；
3. 用户体验设计；
4. 高保真原型预览与确认；
5. 工程规格设计；
6. 受控任务执行；
7. 质量与安全验证；
8. 模拟用户验收；
9. 发布交付；
10. 运行反馈与持续迭代。

模拟用户验收和发布交付不得再次合并：验收证明产品可用，发布批准版本进入目标环境。

### 3.2 AI 工程基础设施

- Context Engineering：让 AI 获得正确、必要、分层和可追溯的项目事实；
- Harness Engineering：让 AI 在阶段、权限、边界、约定和检查关卡内执行；
- Skill Engineering：把真实验证过的方法封装成可重复能力；
- Agent Engineering：定义承担目标的角色、输入输出约定和协作方式；
- Loop Engineering：根据结果、数据和反馈进行观察、评估、纠偏、停止和沉淀。

五类能力贯穿整个生命周期，不是产品阶段。

### 3.3 治理与组织记忆

- 设计决策、版本和事实状态；
- 安全、隐私、权限、知识产权和合规；
- 质量、指标和可观测性；
- 成本、变更、发布和回滚；
- 参考工程对框架的验证反馈。

## 4. 人与 AI 的责任边界

### 人必须负责

- 目标、价值与优先级判断；
- 产品范围和关键取舍；
- 高保真体验确认；
- 敏感数据、权限和高风险变更审批；
- 验收豁免、发布和回滚决定；
- 最终产品、工程和社会责任；
- 专有许可、商业授权、仓库可见性和其他法律治理选择。

### AI 可以负责

- 检索、分析、归纳和方案比较；
- 文档、原型、代码、配置、测试和报告生成；
- 任务拆解和受控工具执行；
- 自动检查、测试、问题定位和候选修复；
- 数据、反馈、失败和成本分析；
- 将验证过的经验整理为候选 Context、模板、检查关卡或 Skill。

AI 不得用“已完成”“应该没问题”“高置信度”等主观判断代替验证证据和人工批准。

## 5. 工作顺序

任何实质性新增或修改均遵循：

```text
定位问题
→ 查阅 LICENSE、宪法、术语规范、全局模型、版本规范和设计决策
→ 读取当前项目 Context 和阶段 Context
→ 确认唯一所属模块和事实源
→ 明确目标、输入、输出、边界、风险与验证
→ 建立或更新任务 Context Pack
→ 修改
→ 自检和运行验证
→ 更新关联专题文档、导航、项目 Context 和版本状态
→ 记录需要长期保留的决策或经验
```

禁止：

- 未查阅关联文档就新增平行概念；
- 建立与现有权威目录同义的平行目录；
- 为追求完整感而批量建设没有参考工程验证的 Skill、脚本或平台；
- 将工具、Agent、Skill、流程、生命周期阶段和组织角色混用；
- 只更新 README，不更新作为事实来源的专题文档；
- 让关键结论只存在于对话、Issue、图片或提交信息中；
- 用局部实现静默改变生命周期、核心原则或责任边界；
- 将内部开发里程碑误写成发布版本；
- 将历史报告中的当时结论误当成当前运行状态；
- 由 Agent 自行选择、更换、放宽或移除专有许可证；
- 将仓库内容复制到未授权仓库、外部服务、训练数据、公共文档或第三方系统；
- 接受未经维护者书面邀请的外部贡献。

## 6. Context 与记忆规则

仓库和受控知识资产是项目长期记忆的事实来源，模型会话不是。

信息优先级：

1. `LICENSE` 和当前有效且未被替代的法律治理决策；
2. 当前有效且未被替代的设计决策；
3. 当前版本宪法、专题文档和正式约定；
4. 当前项目 Context、阶段 Context、Schema、高保真确认和验证记录；
5. 仓库级 Agent 指令；
6. 已批准的任务 Context Pack；
7. 历史任务、历史验证和历史发布报告；
8. 临时对话和模型推断。

历史文件允许保留当时版本、术语和判断，但必须明确标记为历史快照，不得作为当前运行状态的单一依据。

发生冲突时不得静默选择，必须指出：

- 冲突的事实源；
- 各自版本、状态和适用范围；
- 建议保留、替代或归档的结论；
- 是否阻塞当前任务；
- 是否需要新的设计决策。

以下变化必须记录设计决策：

- 框架核心模型或宪法变化；
- 模块边界和单一事实源变化；
- 关键术语含义变化；
- 生命周期阶段增删、拆分或合并；
- 核心检查关卡、权限或责任归属变化；
- 平台绑定或重大技术选择；
- 自动化等级和高风险人工介入点变化；
- 发布版本和开发里程碑语义变化；
- 专有许可、外部授权、仓库可见性或知识产权边界变化。

## 7. 版本、状态和成熟度规则

### 7.1 发布与开发

- 新增一组达到退出标准且经过真实验证的能力增加次版本。

### 7.2 Context 版本字段

必须区分：

- `context_pack_version`；
- `project_context_pack_version`；
- `stage_context_pack_version`；
- `task_context_pack_version`；
- `stable_release`；
- `target_release`；
- `source_version`；
- `source_commit`；
- `baseline_commit`。

禁止使用含义不清的通用 `version` 表达多个概念。

### 7.3 机器状态

YAML 和自动检查关卡使用统一英文机器值，文档正文可以显示中文：

- 事实源：`draft`、`active`、`superseded`、`archived`、`rejected`；
- 阶段：`not_started`、`preparing`、`active`、`in_review`、`approved`、`blocked`、`stopped`；
- 任务：`draft`、`pending_confirmation`、`ready`、`in_progress`、`verifying`、`pending_human_approval`、`completed`、`blocked`、`cancelled`；
- 成熟度：`candidate`、`single_project_validated`、`cross_project_validated`、`stable`、`deprecated`。

## 10. 图示规范

复杂概念优先采用“真实图片资产 + Mermaid + 文字解释”三层表达。

- 真实图片使用仓库内 SVG/PNG，不依赖易失效外链；
- ChatGPT 生成图片用于快速理解和传播，必须标明正式定义的权威来源；
- Mermaid 用于可维护的结构、流程、状态和关系图；
- 每张图只表达一个核心问题；
- 图下必须解释图要证明什么；
- 图片路径、Markdown 链接和 Mermaid 必须经过检查；
- 高保真产品预览是生命周期中的正式检查关卡，不等同于框架宣传图。
