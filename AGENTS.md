# AGENTS.md

## 1. 项目身份

本仓库是 **AI 产品工程框架（AI Product Engineering Framework）**，不是 Prompt 合集、代码模板堆积或某个厂商的专属配置仓库。

项目目标是建立一套开放、跨平台、可验证的产品工程方法，使 Claude Code、Codex、Kimi、GLM 等 AI Agent 能够在人的目标与判断下，受控参与产品定义、体验设计、高保真预览、工程实现、质量验证、发布反馈和持续演进。

当前版本：`v0.1.0`。

当前阶段目标：**先建立完整框架，再逐个模块细化和实现。**

## 2. 不可破坏的核心模型

框架由三个平面组成，禁止将它们重新混成一条扁平“十层流水线”。

### 2.1 产品交付生命周期

1. 战略与价值验证
2. 产品定义
3. 用户体验设计
4. 高保真预览与确认
5. 工程规格设计
6. 受控任务执行
7. 质量与安全验证
8. 模拟用户验收与发布
9. 运行反馈与持续迭代

### 2.2 AI 工程基础设施

- Context Engineering：让 AI 理解项目。
- Harness Engineering：让 AI 在边界内执行并接受验证。
- Skill Engineering：把方法封装成可复用能力。
- Agent Engineering：定义角色、契约和协作。
- Loop Engineering：根据结果、数据和反馈持续修正。

### 2.3 治理与组织记忆

- 设计决策
- 版本与变更
- 安全与权限
- 指标与成本
- 框架验证和参考工程反馈

## 3. 人与 AI 的责任边界

### 人必须负责

- 目标、价值与优先级判断
- 产品范围和关键取舍
- 高保真体验确认
- 高风险变更审批
- 发布与最终验收责任

### AI 可以负责

- 分析、归纳与方案生成
- 文档、原型和工程产物生成
- 任务拆解和受控执行
- 自动检查、测试和问题定位
- 数据与反馈分析

AI 不得用“已完成”“应该没问题”等主观判断代替可验证证据。

## 4. 工作顺序

任何实质性新增或修改均遵循：

```text
定位问题 → 查阅框架与决策 → 确认所属模块 → 明确输入输出 → 修改 → 自检 → 更新关联文档 → 记录重要决策
```

禁止：

- 未查阅关联文档就新增平行概念；
- 为追求完整感而提前建设没有验证场景的 Skill、脚本或平台；
- 将工具、Agent、Skill、流程和角色混用；
- 只更新 README，不更新作为事实来源的专题文档；
- 让关键结论只存在于对话、Issue 或提交信息中。

## 5. Context 与记忆规则

仓库文件是项目长期记忆的事实来源，模型会话不是。

信息优先级：

1. 已批准的设计决策；
2. 当前版本专题文档；
3. `AGENTS.md`；
4. `README.md`；
5. 任务上下文；
6. 对话中的临时讨论。

发生冲突时，不得静默选择。应指出冲突，优先遵从已批准决策，并在需要时新增或更新设计决策。

以下变化必须记录设计决策：

- 框架核心模型变化；
- 模块边界变化；
- 关键术语含义变化；
- 生命周期阶段增删；
- 核心门禁或责任归属变化；
- 平台绑定或重大技术选择。

## 6. 文档与命名规范

- `README.md`、`AGENTS.md`、`CHANGELOG.md` 等通用规范文件保留英文文件名。
- 其他目录和文档优先使用清晰、稳定的中文名称。
- 正文以中文为主，必要的行业术语保留英文并首次解释。
- 避免为了“国际化”复制一套内容相同的英文目录。
- 不使用 `Demo`，统一使用“样例”“参考工程”或“验证工程”。
- 提交信息使用中文，建议格式：`新增：...`、`完善：...`、`修复：...`、`重构：...`、`发布：...`。

## 7. 图示规范

复杂概念优先采用“真实图片资产 + Mermaid + 文字解释”三层表达。

- 真实图片优先使用仓库内 SVG/PNG，不依赖随时可能失效的外链。
- Mermaid 用于可维护的结构、流程、状态和关系图。
- 每张图只表达一个核心问题。
- 图下必须解释图要证明什么，不能只放图不解释。
- 高保真产品预览是生命周期中的正式门禁，不等同于框架宣传插图。

## 8. v0.1 范围约束

v0.1 必须完成：

- 愿景、边界、场景和期望；
- 全局模型和生命周期；
- 角色体系；
- Context、Harness、Skills、Agents、Loop 的定义和关系；
- 决策记录、Roadmap、模板入口和参考工程要求。

v0.1 暂不实现：

- 自动 Agent 编排平台；
- 完整 Skill 库；
- 通用 IDE；
- 无人值守发布系统；
- 未经参考工程验证的企业级治理平台。

## 9. 交付要求

每次完成任务必须说明：

1. 修改了哪些文件；
2. 解决了什么问题；
3. 是否改变核心模型；
4. 验证方式和结果；
5. 尚未解决的风险；
6. 是否需要新增设计决策。
