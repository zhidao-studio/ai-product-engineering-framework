# AGENTS.md

## 1. 项目身份

本仓库是 **AI 产品工程框架（AI Product Engineering Framework）**，不是 Prompt 合集、代码模板堆积、Skill 收藏库或某个厂商的专属配置仓库。

项目目标是建立一套开放、跨平台、可验证的产品工程方法，使 Claude Code、Codex、Kimi、GLM 等 AI Agent 能够在人的目标与判断下，受控参与价值验证、产品定义、体验设计、高保真预览、工程实现、质量验证、用户验收、发布反馈和持续演进。

- 当前稳定版本：`v0.1.0`；
- 当前建设阶段：`v0.2 / Context 可执行化`；
- 总体顺序：**先建立完整框架，再逐个模块细化、实现和验证。**

## 2. 修改前必须读取的事实源

任何实质性修改前，按以下顺序读取：

1. 与任务相关的已批准设计决策；
2. `01_框架定义/AI产品工程框架愿景与定位.md`；
3. `01_框架定义/适用场景与期望.md`；
4. `01_框架定义/AI产品工程核心原则.md`；
5. `01_框架定义/AI产品工程边界声明.md`；

8. 本文件和任务 Context Pack。

README 用于快速理解和导航，不替代专题文档与设计决策。

## 3. 不可破坏的核心模型

框架由三个平面组成，禁止将它们重新混成一条扁平的“十几层流水线”。

### 3.1 产品价值生命周期

根据 `DEC-005`，统一为十个阶段：

1. 战略与价值验证；
2. 产品定义；
3. 用户体验设计；
4. 高保真原型预览与确认；
5. 工程规格设计；
6. 受控任务执行；
7. 质量与安全验证；
8. 模拟用户验收；
9. 发布交付；
10. 运行反馈与持续迭代。

模拟用户验收和发布交付不得再次合并：验收证明产品可用，发布批准版本进入目标环境。

### 3.2 AI 工程基础设施

- Context Engineering：让 AI 获得正确、必要、分层和可追溯的项目事实；
- Harness Engineering：让 AI 在阶段、权限、边界、契约和门禁内执行；
- Skill Engineering：把真实验证过的方法封装成可重复能力；
- Agent Engineering：定义承担目标的角色、输入输出契约和协作方式；
- Loop Engineering：根据结果、数据和反馈进行观察、评估、纠偏、停止和沉淀。

这五类能力贯穿整个生命周期，不是产品阶段。

### 3.3 治理与组织记忆

- 设计决策和事实版本；
- 安全、隐私、权限和合规；
- 质量、指标和可观测性；
- 成本、变更、发布和回滚；
- 参考工程对框架的验证反馈。

## 4. 人与 AI 的责任边界

### 人必须负责

- 目标、价值与优先级判断；
- 产品范围和关键取舍；
- 高保真体验确认；
- 敏感数据、权限和高风险变更审批；
- 验收豁免、发布和回滚决定；
- 最终产品、工程和社会责任。

### AI 可以负责

- 检索、分析、归纳和方案比较；
- 文档、原型、代码、配置、测试和报告生成；
- 任务拆解和受控工具执行；
- 自动检查、测试、问题定位和候选修复；
- 数据、反馈、失败和成本分析；
- 将验证过的经验整理为候选 Context、模板、门禁或 Skill。

AI 不得用“已完成”“应该没问题”“高置信度”等主观判断代替验证证据和人工批准。

## 5. 工作顺序

任何实质性新增或修改均遵循：

```text
定位问题
→ 查阅宪法、全局模型和设计决策
→ 确认唯一所属模块和事实源
→ 明确目标、输入、输出、边界、风险与验证
→ 修改
→ 自检和运行验证
→ 更新关联专题文档与导航
→ 记录需要长期保留的决策或经验
```

禁止：

- 未查阅关联文档就新增平行概念；
- 建立与现有权威目录同义的平行目录，例如同时存在“全局模型”和“核心模型”；
- 为追求完整感而批量建设没有参考工程验证的 Skill、脚本或平台；
- 将工具、Agent、Skill、流程、生命周期阶段和组织角色混用；
- 只更新 README，不更新作为事实来源的专题文档；
- 让关键结论只存在于对话、Issue、图片或提交信息中；
- 用局部实现静默改变生命周期、核心原则或责任边界。

## 6. Context 与记忆规则

仓库和受控知识资产是项目长期记忆的事实来源，模型会话不是。

信息优先级：

1. 当前有效且已批准的设计决策；
2. 当前版本宪法和专题文档；
3. 正式契约、Schema、原型和验证记录；
4. 仓库级 Agent 指令；
5. 当前任务 Context Pack；
6. 临时对话和模型推断。

发生冲突时不得静默选择，必须指出：

- 冲突的两个事实源；
- 各自版本和适用范围；
- 建议保留、替代或归档的结论；
- 是否需要新的设计决策。

以下变化必须记录设计决策：

- 框架核心模型或宪法变化；
- 模块边界和单一事实源变化；
- 关键术语含义变化；
- 生命周期阶段增删、拆分或合并；
- 核心门禁、权限或责任归属变化；
- 平台绑定或重大技术选择；
- 自动化等级和高风险人工介入点变化。

## 7. 文档与命名规范

- `README.md`、`AGENTS.md`、`CHANGELOG.md` 等通用规范文件保留英文文件名；
- 其他目录和文档优先使用清晰、稳定的中文名称；
- 正文以中文为主，必要行业术语保留英文并首次解释；
- 不为了“国际化”复制内容相同的英文目录；
- 不使用 `Demo`，统一使用“样例”“参考工程”或“验证工程”；
- 一个概念只能有一个当前权威文档，其他文件通过链接引用；
- 删除或替代文档前必须确认内部链接和设计决策是否同步；
- 提交信息使用中文，例如：`新增：...`、`完善：...`、`修复：...`、`重构：...`、`发布：...`。

## 8. 图示规范

复杂概念优先采用“真实图片资产 + Mermaid + 文字解释”三层表达。

- 真实图片使用仓库内 SVG/PNG，不依赖易失效外链；
- ChatGPT 生成的图片用于快速理解和传播，必须标明正式定义的权威来源；
- Mermaid 用于可维护的结构、流程、状态和关系图；
- 每张图只表达一个核心问题；
- 图下必须解释图要证明什么，不能只放图不解释；
- 图片路径、Markdown 链接和 Mermaid 必须经过检查；
- 高保真产品预览是生命周期中的正式门禁，不等同于框架宣传图。

- 愿景、定位、适用场景、核心原则和边界；
- 三平面全局模型和十阶段生命周期；

### v0.2 当前子阶段：Context 可执行化

本阶段优先建设：

- Context 分层和单一事实源规范；
- 项目 Context Pack；
- 阶段 Context 和任务 Context Pack；
- Context 装配、冲突、过期和缺失处理；
- 决策与经验如何回写长期记忆；
- Context 完整性检查清单和候选模板。

暂不提前建设：

- 自动 Agent 编排平台；
- 完整 Skill 市场；
- 通用 IDE；
- 无人值守生产发布系统；
- 未经参考工程验证的企业级治理平台。

## 10. 交付要求

每次完成任务必须说明：

1. 修改了哪些文件；
2. 解决了什么问题；
3. 是否改变核心模型或事实源；
4. 验证方式和结果；
5. 尚未解决的风险；
6. 是否新增、替代或归档设计决策；
7. 下一步应进入哪个模块，而不是泛化为“继续完善”。
