# Context 完整性检查清单

> 本清单用于人工评审、Agent 自检和后续自动检查关卡。它检查 Context 是否足以支持受控执行，不评价具体实现是否正确。

中文术语遵循：[术语与易懂表达规范](../01_框架定义/术语与易懂表达规范.md)。

## 1. 使用方式

每项记录：

- `通过`：证据明确；
- `不通过`：缺失、冲突或不符合要求；
- `不适用`：说明原因；
- `有条件通过`：必须写明风险、责任人和截止条件。

高风险任务中，任何 P0 项不通过，任务不得进入执行。

## 2. P0：任务启动必备项

### 2.1 目标与归属

- [ ] 任务能够追溯到用户、业务或工程问题；
- [ ] 任务所属项目和十阶段生命周期阶段明确；
- [ ] 期望结果和明确不做事项清楚；
- [ ] 项目负责人、执行者、验证者和人工批准人明确。

### 2.2 权威事实

- [ ] 已识别项目 Context Pack；
- [ ] 已识别当前阶段 Context；
- [ ] 关键产品、设计和工程事实有权威路径；
- [ ] 关键事实状态为 `active`；
- [ ] 关键事实固定了来源版本、提交或确认日期；
- [ ] 已批准设计决策已被纳入；
- [ ] 没有未处理的 P0 事实冲突。

### 2.3 版本、里程碑与工作段

- [ ] `stable_release` 与 VERSION、CHANGELOG 当前稳定版本一致；
- [ ] `target_release` 与 Roadmap 目标版本一致；
- [ ] 开发里程碑没有被写成发布版本；
- [ ] A1/A2 等工作段没有被写成版本或独立里程碑；
- [ ] 使用 `context_pack_version`、`project_context_pack_version`、`stage_context_pack_version` 或 `task_context_pack_version`，没有含义不清的通用 `version`；
- [ ] 事实源、阶段、任务和成熟度状态使用统一机器枚举；
- [ ] 候选模板没有被描述为 `stable`。

### 2.4 当前执行状态

- [ ] 项目长期状态与当前执行状态已经区分；
- [ ] 当前工作段明确；
- [ ] 当前状态为 `blocked` 时，阻塞原因和解除条件明确；
- [ ] 当前下一步由项目 Context 明确维护；
- [ ] README、AGENTS 和 Roadmap 没有创造不同的当前状态；
- [ ] 历史 TASK、验证或发布报告没有覆盖当前状态。

### 2.5 执行边界

- [ ] 允许修改的目录、文件或系统明确；
- [ ] 禁止修改范围明确；
- [ ] 是否允许新增文件、依赖和配置明确；
- [ ] 工具、环境和权限范围明确；
- [ ] 高风险操作有人工确认点。

### 2.6 约定与不变量

- [ ] API、Schema、权限或用户行为约定已引用；
- [ ] 必须保持的不变量明确；
- [ ] 上下游依赖状态明确；
- [ ] 前置条件满足，任务状态可以进入 `ready`。

### 2.7 验收与停止

- [ ] 验收断言可判断；
- [ ] 静态、运行和用户体验验证方式明确；
- [ ] 不能执行的验证有原因和替代证据；
- [ ] 最大重试、停止和人工升级条件明确；
- [ ] 失败和回滚路径明确。

### 2.8 安全与敏感信息

- [ ] Context 标明敏感级别；
- [ ] 密钥、凭据和生产个人数据未直接写入 Pack；
- [ ] 敏感样本已脱敏或使用合成数据；
- [ ] 受控工具访问的目的和权限明确；
- [ ] 使用的 Agent、模型和环境符合数据边界。

## 3. P1：Context 质量项

### 3.1 正确性

- [ ] 摘要没有改变权威来源原意；
- [ ] 事实与当前代码、约定、高保真和运行状态一致；
- [ ] 模型推断和已确认事实被明确区分；
- [ ] 机器状态与中文显示含义一致；
- [ ] 当前状态与历史快照已经区分。

### 3.2 完整性

- [ ] 当前任务所需产品背景完整；
- [ ] 用户流程、页面和状态信息完整；
- [ ] 工程架构、数据、接口和环境信息完整；
- [ ] 风险、限制、验证和回写要求完整；
- [ ] 当前工作段、阻塞和下一步完整。

### 3.3 相关性与最小化

- [ ] 直接提供的信息与当前任务相关；
- [ ] 详细资料采用按需链接；
- [ ] 已替代、无关和实验草稿已排除；
- [ ] 项目 Context 没有复制全部宪法和专题正文；
- [ ] README、AGENTS、Roadmap 没有重复维护完整当前状态；
- [ ] 没有通过删除边界、风险或验收来压缩 Context。

### 3.4 新鲜度

- [ ] 最近确认时间可见；
- [ ] 复核或失效触发条件明确；
- [ ] 疑似过期内容已标记；
- [ ] 被替代内容不会参与新任务装配；
- [ ] 当前状态变化后项目 Context 已回写。

### 3.5 可追溯性

- [ ] 每项关键事实有来源；
- [ ] 来源有责任人或批准记录；
- [ ] Context Pack 版本可追溯；
- [ ] 项目、阶段和任务 Pack 之间的版本引用完整；
- [ ] 项目 Context 有 `source_commit`；
- [ ] 阶段或任务有 `baseline_commit` / `source_commit`；
- [ ] 基线字段含义和更新时间明确。

## 4. P1：阶段 Context 检查

- [ ] 阶段名称与十阶段生命周期一致；
- [ ] 上一阶段退出证据完整；
- [ ] 本阶段要减少的不确定性明确；
- [ ] 必须产物、责任人和状态明确；
- [ ] 阶段状态使用统一枚举；
- [ ] 阶段长期状态与当前工作段执行状态已区分；
- [ ] 未决事项有阻塞判断、决策人和截止条件；
- [ ] 阶段退出检查关卡可判断；
- [ ] 失败时退回路径明确；
- [ ] 单个任务完成不会被误判为阶段完成。
- [ ] 项目 Context、阶段索引和阶段文件指向同一当前阶段；
- [ ] 阶段文件具有 `source_commit` 和 `last_verified_at`；
- [ ] 阶段文件列出任务、人工批准、PR 和版本变化等复核触发；
- [ ] 过期条件可判断，触发后不会继续参与新任务装配；
- [ ] 新旧阶段通过 `supersedes` / `superseded_by` 形成替代关系。

## 5. P1：历史快照检查

- [ ] 历史 TASK、验证和发布报告顶部标明“历史快照”；
- [ ] 历史文件链接当前项目 Context或后续修正任务；
- [ ] 历史文件保留原始版本、术语、状态和证据；
- [ ] 历史文件中的“下一步”没有被自动当成当前下一步；
- [ ] 历史结论与当前状态冲突时，当前项目 Context 优先；
- [ ] 当前文件没有为了统一表述而篡改历史证据。

## 6. P1：回写准备检查

- [ ] 任务 Pack 预先指定了可能的长期回写位置；
- [ ] 任务结果会区分临时结果和长期事实；
- [ ] 失败原因将按 Context、Harness、Skill、Agent、工具或产品假设分类；
- [ ] 自动记忆只作为候选；
- [ ] 新经验有统一成熟度状态；
- [ ] 重大变化会形成设计决策；
- [ ] 旧事实会被标记为 `superseded` 或 `archived`；
- [ ] 稳定版本和目标版本变化会同步 CHANGELOG、Roadmap、README、AGENTS 和项目 Context；
- [ ] 当前工作段或阻塞变化会同步项目 Context 和阶段 Context。

## 7. Context Pack 评分建议

评分不能替代 P0 检查关卡，但可以观察成熟度：

| 维度 | 权重 |
|---|---:|
| 正确性 | 20 |
| 完整性 | 20 |
| 相关性与最小化 | 15 |
| 新鲜度 | 10 |
| 可追溯性 | 15 |
| 安全性 | 15 |
| 回写准备 | 5 |

建议解释：

- `90–100`：可以进入受控执行；
- `75–89`：补充 P1 项后执行；
- `60–74`：Context 风险较高，应重新装配；
- `<60`：不得执行。

任何 P0 项不通过时，无论总分多少均不得执行。

历史评分只证明当时基线，不自动代表当前仓库仍保持同一分数。发生稳定版本、项目 Pack 或核心状态变化后，应重新检查或明确评分仍是历史证据。

## 9. 反模式

- 只检查文档是否存在，不检查内容状态和版本；
- 用总分掩盖 P0 缺失；
- 检查者和执行者都是同一个 Agent，且无独立证据；
- “有条件通过”没有责任人和截止条件；
- 敏感信息检查只写“无风险”；
- 检查后不更新 Context Pack 和事实源；
- 每次都人工检查同样问题，却不形成检查关卡；
- 将里程碑、工作段、Pack 版本和发布版本混写；
- 中文状态和机器状态表达不一致；
- 项目长期 `active` 被错误解释为当前执行未阻塞；
- 使用历史 `partial_pass` 覆盖当前 `not_started`；
- 把旧评分直接当作当前全仓成绩。
