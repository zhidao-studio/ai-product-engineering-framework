# 项目 Context Pack 规范

> 项目 Context Pack 是项目长期事实的入口和导航，使新的人员、Agent、模型或会话能够快速恢复一致的项目认知。

中文术语遵循：[术语与易懂表达规范](../01_框架定义/术语与易懂表达规范.md)。

## 1. 定义

项目 Context Pack 不是把所有项目文档复制到一个大文件中，而是：

- 汇总项目最重要的当前事实；
- 指向权威产品、设计、工程和治理资产；
- 标明状态、版本、责任人和敏感边界；
- 说明稳定发布、目标发布、当前阶段和已知风险；
- 说明进入具体任务前还需要读取什么。

它是**入口和装配清单**，不是替代专题文档的第二套事实源。

## 2. 适用时机

以下情况必须建立或更新项目 Context Pack：

- 新项目进入正式产品定义；
- 新的 AI Agent 或人员接手项目；
- 跨会话、跨模型或跨平台继续工作；
- 产品、设计、架构或关键约定发生变化；
- 项目进入新的生命周期阶段；
- 稳定版本、目标版本或开发里程碑变化；
- 发生重大事故、回滚或方向调整；
- 参考工程需要验证 Framework 的记忆连续性。

## 3. 推荐结构

```text
项目Context/
├── README.md                    # 项目 Context Pack 入口
├── 01_项目目标与边界.md
├── 02_产品与业务规则索引.md
├── 03_体验与高保真入口.md
├── 04_工程架构与约定索引.md
├── 05_环境权限与工具.md
├── 06_验证发布与反馈.md
├── 07_设计决策索引.md
└── 08_当前状态风险与下一步.md
```

小型项目可以合并为一个 Markdown 文件，但必须保持必要信息块完整。

## 4. 元数据

推荐头部：

```yaml
project_context_id: PROJ-CONTEXT-001
project: 项目名称
status: active
context_pack_version: 0.2-A.2
owner: 项目负责人
current_stage: 工程规格设计
stable_release: v0.1.4
target_release: v0.2.0
current_milestone: A / Context 可执行化
working_branch: main
last_verified_at: 2026-07-12
source_commit: abc1234
sensitivity: internal
```

字段要求：

- `status` 使用统一机器枚举；
- `context_pack_version` 只表示 Pack 自身版本；
- `stable_release` 表示当前稳定发布；
- `target_release` 表示当前开发目标；
- `current_milestone` 不是发布版本；
- `source_commit` 固定 Pack 对应的仓库基线。

不得使用含义不清的通用 `version` 字段。

## 5. 必要内容

### 5.1 项目身份

- 项目名称和一句话定位；
- 目标用户和核心问题；
- 当前稳定版本、目标版本和开发里程碑；
- 当前生命周期阶段；
- 产品负责人、体验确认人和工程责任人；
- 默认分支、基线提交和 Agent 规则入口。

### 5.2 目标、指标与停止条件

- 当前版本要达成的产品目标；
- 可观察的成功指标；
- 时间、成本和资源约束；
- 明确不做内容；
- 继续、调整或停止条件。

### 5.3 产品与业务事实入口

至少链接：

- PRD 或产品定义；
- 不做清单；
- 用户角色；
- 业务规则和口径；
- 验收断言；
- 已批准产品决策。

### 5.4 体验与高保真入口

至少链接：

- 用户主流程；
- 页面和功能范围；
- 正常、加载、空、错误、权限和边缘状态；
- 高保真原型或图片；
- 人工确认记录；
- 视觉和交互规范。

### 5.5 工程事实入口

至少链接：

- 系统总览和架构；
- 模块边界；
- API 接口约定；
- 数据模型和 Schema；
- 依赖和版本；
- 环境、部署和运行方式；
- 安全、权限和隐私要求；
- 技术设计决策。

### 5.6 AI 执行规则

- `AGENTS.md`、`CLAUDE.md` 或平台适配规则；
- 允许和禁止使用的工具；
- 默认修改边界；
- 依赖新增规则；
- 测试、验证和输出要求；
- 高风险人工确认点；
- 不同平台能力差异。

### 5.7 当前状态

- 当前稳定版本、目标版本、里程碑和生命周期阶段；
- 已完成、进行中、阻塞和待决事项；
- 当前风险和技术债务；
- 最近验证结果；
- 最近发布状态；
- 下一步进入哪个模块或任务。

### 5.8 事实源索引

| 名称 | 来源 | 来源版本或提交 | 状态 | 责任人 | 最近确认 |
|---|---|---|---|---|---|
| 产品范围 | 链接 | v1.2 | active | 产品负责人 | 日期 |
| 高保真主流程 | 链接 | Prototype 3 | active | 体验确认人 | 日期 |
| OpenAPI | 链接 | v1.4 | active | 后端责任人 | 日期 |

### 5.9 敏感信息声明

必须说明：

- 哪些信息不进入模型；
- 哪些内容已经脱敏；
- 哪些信息只能通过受控工具查询；
- 使用模型、区域或环境限制；
- 凭据和密钥管理入口，但不得写入明文。

## 6. 更新触发

项目 Context Pack 在以下变化后必须更新：

- 目标、范围或不做清单变化；
- 高保真主流程重新确认；
- 架构、API、Schema 或依赖变化；
- 项目进入新阶段；
- 稳定发布、目标发布或里程碑变化；
- 责任人或平台变化；
- 发布、回滚或重大故障；
- 新设计决策生效；
- 事实源被替代或归档。

任务级局部修改不一定更新项目 Pack，但如果影响长期事实，必须回写。

## 7. 验收标准

一个陌生但具备相应权限的 Agent 或人员，只读取项目 Context Pack 和它明确链接的必要文件后，应能回答：

1. 项目为什么存在；
2. 当前为谁解决什么问题；
3. 当前做什么、不做什么；
4. 用户如何使用，在哪里确认高保真；
5. 系统如何组织，关键约定在哪里；
6. 当前稳定版本、目标版本和里程碑是什么；
7. 当前处于哪个生命周期阶段；
8. AI 默认能改什么、不能改什么；
9. 如何构建、测试、验证和发布；
10. 当前有哪些风险、阻塞和待决策；
11. 具体任务开始前还需装配哪些 Context。

不能回答上述问题，说明项目 Context Pack 不完整。

## 8. 反模式

- 把全部文档复制成超长单文件；
- 只描述技术栈，不描述产品目标和用户；
- 只写目录，不标当前有效版本；
- 把 `context_pack_version`、`stable_release` 和 `target_release` 混成一个 `version`；
- 将内部里程碑误写成发布版本；
- 链接过期或指向未批准草稿；
- 在 Pack 中放入密钥和生产个人数据；
- 把任务临时信息长期写入项目事实；
- 项目变化后只更新 README，不更新 Pack；
- 将 Pack 当作永久不变的静态说明。
