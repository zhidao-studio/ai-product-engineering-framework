# 设计决策记录规范

## 1. 目的

设计决策记录用于保存“为什么这样设计”，防止关键结论随对话、人员、模型或任务重点变化而丢失。

## 2. 需要记录的情况

- 核心模型和模块边界变化；
- 生命周期阶段增删或责任改变；
- 关键术语重新定义；
- 重大技术、平台或数据选择；
- 关键安全、权限和成本策略；
- 多种合理方案之间的长期取舍；
- 撤销或替代已有决策。

普通文字修正、格式调整和无争议的小改动不需要单独决策记录。

## 3. 状态

- `提议中`：仍在讨论；
- `已接受`：当前有效；
- `已废弃`：不再建议使用；
- `已替代`：由新的决策取代，并必须给出编号。

## 4. 标准模板

```markdown
# DEC-XXX：决策标题

- 状态：提议中 / 已接受 / 已废弃 / 已替代
- 日期：YYYY-MM-DD
- 决策人：
- 关联文档：

## 背景
为什么需要做决定？

## 问题
需要解决的核心冲突是什么？

## 备选方案
列出主要方案及其优缺点。

## 决定
最终采用什么方案？

## 原因
为什么选择它？

## 影响
需要修改哪些文档、模板、Skills、门禁或参考工程？

## 验证
如何知道该决定有效？
```

## 5. 维护规则

- 已接受决策不得静默重写；
- 改变结论时新增决策并标记替代关系；
- 专题文档只写当前有效规则，决策记录保留历史原因；
- `11_设计决策/README.md` 维护索引和状态；
- 重要提交必须同步检查是否需要决策记录。
