## 1. 目标与价值来源

### 问题

Framework 已建立 Context 规范和候选模板，但尚未用自身项目验证。若框架不能管理自己的事实、阶段和任务，就不应直接要求业务项目采用。

### 目标

建立本仓库的项目 Context Pack、v0.2-A 阶段 Context、任务记录和完整性检查，使新的人员或 Agent 能够不依赖历史会话恢复当前项目认知。

### 不做

- 不选择业务参考工程；
- 不建设自动 Context 装配器；
- 不建设 Harness 脚本；
- 不把自应用结果标记为跨项目稳定结论。

## 8. 验证方式

| 验证 | 方法 | 预期结果 |
|---|---|---|
| 路径验证 | 逐一读取项目 Context 中的核心链接 | 链接存在且指向当前权威文件 |
| 口径验证 | 对比 README、AGENTS、全局模型、DEC-005 | 均使用十阶段生命周期 |
| 决策验证 | 读取 DEC-001 至 DEC-007 索引 | 重要变化可恢复 |
| 重复检查 | 检查已知重复目录和文件 | 不再存在平行事实源 |
| Context 检查 | 使用 `Context完整性检查清单.md` | P0 全部通过 |
| 人工复核 | 项目维护者阅读 Pack | 能理解当前状态并批准下一步 |

## 9. 风险与停止条件

- 如发现新的平行事实源，停止并先收敛目录；
- 如 Pack 需要复制大量正文才能理解，回到项目 Pack 规范调整；
- 如核心链接不存在，修复后重新检查；
- 如需要改变宪法或十阶段模型，必须停止并新建设计决策；
- 最大自动修正轮次：2 次，超过后提交项目维护者判断。
