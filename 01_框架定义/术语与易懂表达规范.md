# 术语与易懂表达规范

> 本规范统一 Framework 面向中文读者的表达。专业概念不降低准确性，但优先使用能够直接理解的中文词语。

## 1. 核心规则

| 旧中文表述 | 当前中文表述 | 含义 |
|---|---|---|
| 契约 | 约定 | 系统、模块、角色或阶段之间必须共同遵守且可以检查的规则 |
| 门禁 | 检查关卡 | 进入下一步之前必须达到的条件；未通过时必须停止、退回或人工处理 |

推荐首次出现时写为：

```text
约定（Contract）
检查关卡（Gate）
```

后续中文正文直接使用“约定”和“检查关卡”。

## 2. 常用表达

| 技术表达 | Framework 中文表达 |
|---|---|
| API Contract | 接口约定 |
| Data Contract | 数据格式约定 |
| Schema Contract | 数据结构约定 |
| Behavioral Contract | 功能行为约定 |
| Contract Test | 约定一致性检查 |
| Quality Gate | 质量检查关卡 |
| Design Gate | 设计确认关卡 |
| Merge Gate | 合并前检查关卡 |
| Release Gate | 发布前检查关卡 |
| Stage Gate | 阶段通过条件或阶段检查关卡 |

## 3. 保留的技术名称

以下内容不强制翻译或改名：

- `OpenAPI`、`Schema`、`Contract`、`Gate` 等正式技术标准或英文概念；
- `GATE-001`、`gate_id`、`contract_checks` 等机器标识、字段名和已有编号；
- 外部平台、工具或协议中的固定名称；
- 已发布历史报告、任务记录和验证证据中的原始表述。

保留技术名称时，面向人的中文说明仍使用“约定”“检查关卡”。例如：

```text
GATE-001：iOS 源代码注册检查关卡
OpenAPI：前后端接口约定的权威文件
```

## 4. 使用原则

1. 中文目录、标题、正文、模板说明和用户可见报告优先使用易懂表述；
2. 不为了替换词语而改变原有工程含义；
3. “约定”必须明确内容、参与方、版本和检查方法；
4. “检查关卡”必须明确通过条件、失败处理和是否需要人工确认；
5. 检查清单只说明检查什么，检查关卡还决定是否允许继续；
6. 历史记录不追溯篡改，通过本规范解释旧称与当前术语的关系；
7. 新增文档不得重新把旧称作为主要中文表述。

## 5. 示例

不推荐：

```text
检查 OpenAPI 契约门禁。
```

推荐：

```text
执行前后端接口约定一致性检查；未通过接口检查关卡时禁止合并。
```
