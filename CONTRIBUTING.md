# 贡献指南

感谢参与 AI Product Engineering Framework。

本项目优先接受能够提高框架可理解性、可执行性、可验证性和真实产品价值的贡献，不以文档、Prompt、Skill 或 Agent 数量作为成熟度标准。

## 1. 贡献前先阅读

按顺序阅读：

1. `AGENTS.md`；
2. `01_框架定义/` 中的愿景、场景、核心原则和边界声明；
3. `02_全局模型/AI产品工程全局框架.md`；
4. 与贡献内容相关的专题文档；
5. `11_设计决策/README.md` 和相关决策；
6. `10_版本演进/版本管理规范.md`。

## 2. 适合贡献的内容

- 修复错误、断链、过期示例和不一致术语；
- 补充真实项目中发现的 Context、Harness、验证或反馈缺口；
- 提供经过验证的模板、门禁、Skill 或平台适配；
- 增加参考工程的验证证据和失败复盘；
- 改善图示、文档导航和非技术人员可理解性；
- 提出有明确问题、备选方案和影响分析的设计决策。

## 3. 不建议直接提交的内容

- 未经真实任务验证的大量 Prompt 或 Skills；
- 与现有权威文档同义的平行目录；
- 只展示最终代码、没有产品目标和验证证据的样例；
- 把某个平台当前能力写成永久框架原则；
- 宣称完全无人化，但没有权限、停止、审计和回滚机制的方案；
- 直接包含密钥、凭据、个人隐私或未授权企业信息的材料。

## 4. 贡献分类

### 文档修订

适用于错别字、链接、图示、术语和不改变核心模型的表达修正。

必须说明：

- 修改文件；
- 原问题；
- 修正依据；
- 是否影响其他文档和版本。

### 框架能力新增

适用于 Context、Harness、Skills、Agents、Loop、模板、门禁或参考工程。

必须说明：

- 服务哪个生命周期阶段和真实问题；
- 依赖什么权威 Context；
- 输入、输出、责任人和边界；
- 验证方式、失败和停止条件；
- 已在哪个真实任务中验证；
- 成熟度是候选、单项目已验证还是跨项目已验证。

### 核心模型变更

涉及宪法、三平面模型、十阶段生命周期、五大基础设施、责任边界或版本语义时，必须先新增设计决策。

## 5. 文件与命名

- `README.md`、`AGENTS.md`、`CHANGELOG.md`、`CONTRIBUTING.md` 等通用文件保留英文文件名；
- 其他目录和文档优先使用清晰中文名称；
- 不使用 `Demo`，统一使用“样例”“参考工程”或“验证工程”；
- 一个概念只保留一个当前权威文档；
- 提交信息使用中文，例如：`修复：...`、`新增：...`、`完善：...`、`重构：...`、`发布：...`。

## 6. 图示要求

复杂概念优先使用：

```text
真实图片或高保真预览
+ Mermaid 工程图
+ Markdown 正式说明
```

图片不得成为唯一事实来源。ChatGPT 生成图片中的文字和关系必须由正式文档或 Mermaid 校验。

## 7. 提交前检查

- [ ] 已阅读相关宪法、专题文档和设计决策；
- [ ] 没有创建同义平行目录；
- [ ] 内部链接和图片路径有效；
- [ ] Mermaid 基础语法可渲染；
- [ ] 版本和状态表达符合版本管理规范；
- [ ] 未把候选能力误写成稳定能力；
- [ ] 未提交敏感信息；
- [ ] 说明了验证方式和结果；
- [ ] 需要回写的 README、AGENTS、CHANGELOG、Roadmap 或索引已同步；
- [ ] 核心变化已经形成设计决策。

## 8. Review 标准

Review 不只判断“内容是否合理”，还要判断：

1. 是否解决真实问题；
2. 是否属于正确模块；
3. 是否与单一事实源一致；
4. 是否有明确输入、输出和责任；
5. 是否可验证；
6. 是否具有安全、权限、成本和停止边界；
7. 是否经过足够的真实验证；
8. 是否应进入当前版本。

## 9. 许可证说明

本仓库当前尚未提交正式 `LICENSE` 文件。许可证属于仓库维护者的独立决定，贡献者不应自行添加或更换许可证。待许可证确定后，本节和根目录文件必须同步更新。
