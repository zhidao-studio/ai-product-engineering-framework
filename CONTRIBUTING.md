# 受邀协作者指南

> 本仓库采用专有闭源许可，不接受未经邀请的外部贡献。只有获得 zhidao-studio 书面邀请并被授予相应仓库权限的协作者，才可以在授权范围内参与。

正式权利边界见根目录 [LICENSE](LICENSE)。参与协作不代表获得本仓库其他内容的使用、复制、分发、商业化或衍生权利。

## 1. 协作前置条件

受邀协作者必须：

1. 获得 zhidao-studio 的明确书面邀请；
2. 仅使用维护者授权的账号、仓库、分支、环境和工具；
3. 阅读 `LICENSE`、`AGENTS.md`、框架宪法、全局模型和相关设计决策；
4. 遵守任务 Context Pack 中的允许与禁止范围；
5. 不向任何未授权个人、组织、模型、知识库、数据集或第三方服务披露仓库内容；
6. 如涉及额外知识产权、保密或商业条款，先完成单独书面协议。

## 2. 不接受的行为

- 未经邀请提交 Issue、Pull Request、Fork 后的改造或外部镜像；
- 将仓库内容用于个人项目、企业内部系统、课程、咨询、商业交付或模型训练；
- 复制、摘录、翻译、改编、分发或创建衍生作品；
- 将仓库内容上传到未授权的 AI 模型、代码助手、搜索服务或知识管理系统；
- 绕过权限、门禁、人工确认或许可证限制；
- 提交密钥、凭据、个人隐私或未授权企业信息。

## 3. 受邀协作内容

受邀协作者可以在任务授权范围内：

- 修复错误、断链、过期示例和不一致术语；
- 补充真实项目中发现的 Context、Harness、验证或反馈缺口；
- 提供经过验证的模板、门禁、Skill 或平台适配；
- 增加参考工程验证证据和失败复盘；
- 改善图示、导航和可理解性；
- 提出包含问题、备选方案和影响分析的设计决策。

任何新增内容必须说明其知识产权来源，确保维护者有权将其纳入本专有仓库。

## 4. 修改前阅读顺序

1. `LICENSE`；
2. `AGENTS.md`；
3. `01_框架定义/` 中的愿景、场景、核心原则和边界声明；
4. `02_全局模型/AI产品工程全局框架.md`；
5. 与任务相关的专题文档；
6. `11_设计决策/README.md` 和相关决策；
7. `10_版本演进/版本管理规范.md`；
8. 当前项目、阶段和任务 Context Pack。

## 5. 修改要求

### 文档修订

必须说明：

- 修改文件；
- 原问题；
- 修正依据；
- 是否影响其他事实源、版本和许可边界；
- 验证方式和结果。

### 框架能力新增

必须说明：

- 服务哪个生命周期阶段和真实问题；
- 依赖什么权威 Context；
- 输入、输出、责任人和执行边界；
- 验证方式、失败和停止条件；
- 已在哪个真实任务中验证；
- 当前成熟度。

### 核心模型或许可变更

涉及框架宪法、三平面模型、十阶段生命周期、五大基础设施、责任边界、版本语义、专有许可或仓库可见性时，必须先形成设计决策，并由 zhidao-studio 批准。

## 6. 文件与命名

- `README.md`、`AGENTS.md`、`CHANGELOG.md`、`CONTRIBUTING.md`、`LICENSE` 等通用文件保留英文文件名；
- 其他目录和文档优先使用清晰中文名称；
- 不使用 `Demo`，统一使用“样例”“参考工程”或“验证工程”；
- 一个概念只保留一个当前权威文档；
- 提交信息使用中文，如 `修复：...`、`新增：...`、`完善：...`、`重构：...`、`发布：...`。

## 7. 提交前检查

- [ ] 已确认自己是受邀且已授权的协作者；
- [ ] 已阅读 LICENSE、AGENTS、相关事实源和设计决策；
- [ ] 修改未超出任务授权范围；
- [ ] 未复制仓库内容到外部或未授权系统；
- [ ] 未创建同义平行目录；
- [ ] 内部链接、图片和 Mermaid 可用；
- [ ] 版本、状态和成熟度表达正确；
- [ ] 未把候选能力写成稳定能力；
- [ ] 未提交敏感信息；
- [ ] 验证结果和回写位置明确；
- [ ] 需要同步的 README、AGENTS、CHANGELOG、Roadmap 和索引已更新。
