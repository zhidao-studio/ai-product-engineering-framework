# 接口、数据与实现一致性规范

> 本规范控制 OpenAPI、Schema、服务端、客户端和错误码之间的可追溯关系。它不假设“文件存在”就等于业务一致，只允许根据实际执行证据得出有限结论。

## 1. 一致性对象

| 对象 | 权威事实源 | 必须对齐的实现 |
|---|---|---|
| HTTP 路径、方法、请求与响应 | 已批准 OpenAPI | Controller、客户端 API、集成测试 |
| 业务字段、类型、是否必填 | 已批准 OpenAPI 与数据字典 | DTO、实体、客户端模型、序列化 |
| 数据表、索引、触发器与迁移 | 已批准 Schema 和数据库规范 | 迁移脚本、实体、Mapper、回滚脚本 |
| 错误码、HTTP 状态和用户提示 | 错误码约定 | 服务端异常、客户端映射、异常测试 |
| 身份、权限和信任边界 | 安全设计与接口约定 | Gateway、服务、客户端、运行验证 |

发生冲突时，不允许由 Agent 自行选择“看起来更新”的文件。必须先确认哪一份是当前批准事实源，再修正其他实现或退回工程规格阶段。

## 2. 最小追溯矩阵

每个跨端业务能力至少记录：

| 能力或字段 | OpenAPI | Schema/数据字典 | 服务端 | 客户端 | 错误码 | 验证 |
|---|---|---|---|---|---|---|
| 示例 | 路径或组件 | 表和字段 | 类或方法 | 模型或调用 | 编码 | 命令与证据 |

矩阵用于发现缺口，不替代实际验证。只有列出路径而没有检查字段含义、运行行为和错误路径时，最多证明“已建立追溯入口”。

## 3. 检查顺序

```mermaid
flowchart LR
    A[确认批准事实源] --> B[建立追溯矩阵]
    B --> C[静态结构检查]
    C --> D[构建与类型检查]
    D --> E[接口与数据运行验证]
    E --> F[客户端真实调用]
    F --> G[异常与错误码验证]
    G --> H[记录结论边界]
```

1. 确认版本、提交和批准状态；
2. 检查路径、方法、字段名、类型、必填、枚举和错误码；
3. 检查迁移、实体、Mapper、DTO 和客户端模型；
4. 运行构建、Schema 应用、接口和客户端调用；
5. 验证至少一个成功路径和关键失败路径；
6. 把未执行项写成 `not_executed`，不得用静态阅读替代。

## 4. 结果状态

| 状态 | 含义 |
|---|---|
| `passed` | 指定对象在指定基线和环境中通过已列出的检查 |
| `conditional_pass` | 已通过部分必要检查，但仍有明确未完成项 |
| `failed` | 已执行检查并出现不满足约定的结果 |
| `blocked` | 因环境、权限、上游或安全条件无法继续 |
| `not_executed` | 尚未执行，不能形成通过结论 |

禁止把 `conditional_pass`、`blocked` 或 `not_executed` 汇总成 `passed`。

## 5. 自动与人工判断边界

确定性检查可以判断：

- OpenAPI 或 JSON 是否可解析；
- 声明的路径、组件、字段或错误码是否存在；
- 生成代码是否有未提交差异；
- Schema 是否能在受控空库应用和回滚；
- 构建、测试和接口脚本的实际退出码。

确定性检查不能单独判断：

- 字段语义是否满足业务；
- 权限模型是否完整；
- 用户提示是否可理解；
- 兼容性取舍是否可接受；
- 生产数据迁移是否安全。

这些内容必须由工程审查、运行证据和人工确认补充。

## 6. 失败处理

- 权威事实源冲突：停止实现，退回工程规格设计；
- 静态不一致：在任务边界内修正后重新检查；
- 运行行为与约定不一致：保留请求、响应、日志和提交，分类为实现或环境问题；
- 需要改变批准接口或数据模型：扩大任务前重新批准；
- 已部署兼容性破坏：停止继续发布，执行已批准回滚或版本兼容方案。
