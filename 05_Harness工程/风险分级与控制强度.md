# 风险分级与控制强度

> 本文定义 AI 产品工程任务的风险等级及最低控制要求。它决定 Harness 要多严格，不代替产品、高保真、发布或法律责任人的人工决定。

## 1. 分级目的

不同任务不应使用同样重量的流程：格式修正不需要生产发布级控制，数据库删除也不能按普通文档任务处理。

风险分级用于决定：

- Context Pack 深度；
- 修改范围精度；
- 是否允许自主执行；
- 需要哪些验证层；
- 自动重试次数；
- 人工批准、发布和回滚要求。

## 2. 判断维度

从以下维度判断任务风险：

| 维度 | 低风险信号 | 高风险信号 |
|---|---|---|
| 用户影响 | 不改变用户行为 | 页面、流程、账号、支付或可用性变化 |
| 数据影响 | 不修改持久化数据 | 迁移、删除、隐私、资金或批量写入 |
| 权限安全 | 不涉及身份和密钥 | 鉴权、授权、签名、密钥或外部暴露 |
| 环境 | 本地和隔离样例 | 测试共享环境、生产或真实外部服务 |
| 可逆性 | 单提交即可完整回退 | 数据不可逆、外部副作用或回滚困难 |
| 变更范围 | 单文件局部变化 | 跨模块、跨仓库、公共约定或基础设施 |
| 规格确定性 | 已有明确批准基线 | 产品、设计、接口或责任仍有歧义 |
| 验证能力 | 可本地自动验证 | 依赖真机、真实用户、生产数据或人工判断 |

任务采用所有维度中的**最高风险等级**。信息不足时不能按低风险处理；不确定性明显时至少提高一级，或先停止补充事实。

## 3. 四级风险

### 3.1 `low`：低风险

典型任务：

- 已确认内容中的错别字和断链；
- 不改变行为的格式整理；
- 单文件、容易回退的静态修正。

最低控制：

- 轻量任务记录；
- 明确目标、允许和禁止范围；
- 至少一次静态检查；
- 记录实际修改和验证结果；
- 不得顺便扩大范围。

### 3.2 `medium`：中风险

典型任务：

- 已有规格下的局部代码实现或缺陷修复；
- 非敏感内部接口调整；
- 影响有限、可完整回滚的配置或测试变化；
- 与已批准高保真对齐的 UI 修正。

最低控制：

- 完整任务 Context Pack；
- 文件或目录级允许、禁止范围；
- 固定代码基线和事实版本；
- 静态检查加构建或相关测试；
- 用户可见变化需要视觉或用户路径验证；
- 最多两轮同类确定性自动修复，仍失败则升级。

### 3.3 `high`：高风险

以下任一项通常至少为高风险：

- 数据库 Schema、迁移、触发器或批量数据；
- 鉴权、授权、会话、签名和内部身份；
- 公共 API、OpenAPI、跨端数据约定；
- 新增或替换依赖；
- 真机权限、文件上传、敏感日志；
- 跨仓库或多个可部署服务；
- 影响正式用户流程的页面、导航或业务规则。

最低控制：

- 项目、阶段和任务 Context 版本全部固定；
- 精确修改白名单和高风险禁止项；
- 执行前确认工程规格、回滚和验证环境；
- 静态、运行和适用的用户体验验证；
- 关键实现和最终结果分别需要人工确认；
- 每类失败最多自动修复一轮，扩大范围前必须重新批准；
- 数据、权限和外部副作用必须有恢复或补偿方案。

### 3.4 `critical`：严重风险

以下任一项通常为严重风险：

- 生产发布、生产数据库和真实用户数据；
- 物理删除、不可逆迁移或大范围覆盖；
- 支付、资金、结算或财务口径；
- 大规模隐私数据导出或权限放开；
- 根密钥、生产证书和核心身份边界；
- 法律、许可或外部公开承诺；
- 故障可能造成重大业务、合规或安全影响。

最低控制：

- 独立任务和独立书面授权；
- 执行人、复核人、批准人职责分离；
- 备份、演练、回滚和停止条件先于执行；
- 对破坏性动作逐步人工确认；
- 禁止自动重试破坏性动作；
- 必须保留完整审计证据；
- 未取得最终人工批准不得发布或关闭任务。

## 4. 控制强度矩阵

| 控制项 | low | medium | high | critical |
|---|---|---|---|---|
| Context | 轻量任务记录 | 完整任务 Pack | 项目+阶段+任务 Pack | 全量 Pack 与独立授权 |
| 修改边界 | 文件或明确局部 | 文件/目录白名单 | 精确白名单与高风险禁区 | 分步骤授权和职责分离 |
| 规格基线 | 当前事实引用 | 固定版本或提交 | 全部关键约定固定 | 独立复核并签署 |
| 静态检查 | 必须 | 必须 | 必须 | 必须并独立复核 |
| 运行验证 | 适用时 | 必须执行相关构建/测试 | 必须覆盖真实依赖链路 | 先演练，再受控执行 |
| 用户验收 | 不适用或抽查 | 用户可见时必须 | 用户流程变化必须 | 发布前必须且职责分离 |
| 自动重试 | 最多 2 轮 | 最多 2 轮 | 每类最多 1 轮 | 破坏性动作 0 轮 |
| 人工批准 | 固有人工节点 | 用户可见或范围变化时 | 执行前和结果后 | 每个关键步骤与最终发布 |
| 回滚 | Git 回退 | 明确代码/配置回退 | 数据和外部副作用方案 | 备份、演练和应急责任人 |

## 5. 不可降级的人工确认

无论风险等级如何，以下事项都不能由 AI 自行批准：

- 产品范围和关键取舍；
- 高保真体验；
- 敏感数据和权限；
- 验收豁免；
- 生产发布和回滚；
- 许可与法律责任。

## 6. 升级与降级规则

必须升级风险：

- 实际修改超过原范围；
- 新增数据库、权限、依赖或生产影响；
- 发现真实数据或敏感信息；
- 原验证环境无法复现；
- 回滚不再可靠；
- 同类失败超过允许次数。

风险降级必须记录：

1. 原等级和拟降等级；
2. 哪些事实消除了风险；
3. 哪些控制仍然保留；
4. 人工批准人和日期。

AI 不得自行降低风险等级。

## 7. 正反例

| 任务 | 建议等级 | 原因 |
|---|---|---|
| 修复 README 断链 | low | 无运行和用户行为变化 |
| 按批准高保真修正按钮间距 | medium | 用户可见但规格已确定、可回退 |
| 新增手机号验证码登录接口 | high | 涉及账号、会话、数据和跨端约定 |
| 修改生产账号表并迁移历史用户 | critical | 真实用户数据和回滚风险 |
| 增加本地开发日志 | medium | 需验证不记录敏感信息 |
| 将生产 Token 输出到日志以便排错 | 禁止执行 | 不是提高风险等级可以接受的方案 |
