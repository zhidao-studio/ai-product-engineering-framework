# 依赖与高风险变更授权规范

> 本文定义新增、升级、替换和移除依赖的最低评估，以及高风险文件或操作的额外授权。依赖文件被修改不等于依赖变更已经被批准。

## 1. 依赖变更类型

- 新增直接或间接依赖；
- 升级、降级或锁定版本；
- 替换已有框架或运行时；
- 移除依赖；
- 修改仓库源、插件、构建脚本或锁文件；
- 启用原本未使用的可选模块或外部服务。

## 2. 依赖准入信息

每次依赖变更必须说明：

| 项目 | 必填内容 |
|---|---|
| 解决的问题 | 为什么现有代码或依赖不能满足 |
| 使用位置 | 哪些模块、目标和环境使用 |
| 版本来源 | 版本、仓库和锁定方式 |
| 维护状态 | 维护者、更新频率、已知停止维护风险 |
| 许可 | 许可证及与项目许可的兼容判断 |
| 安全 | 已知漏洞、权限、网络和数据影响 |
| 平台影响 | 包体、构建、运行时、最低系统和性能 |
| 替代方案 | 不引入、内部实现或其他依赖的比较 |
| 移除成本 | 未来替换或删除需要改动什么 |
| 验证与回滚 | 构建、测试、运行和回退方式 |
| 批准 | 人工批准人、日期和适用范围 |

没有许可证或来源不明的依赖不得引入专有仓库。

## 3. 默认拒绝条件

以下情况默认拒绝：

- 与现有能力重复，但没有替换计划；
- 只为绕过测试、运行或安全验证；
- 维护停止、来源不明或许可证不清；
- 要求扩大敏感数据、设备或生产权限但没有必要性；
- 引入第二套核心鉴权、网络、序列化或数据库框架而没有设计决策；
- 依赖动态未固定版本或不可复现来源；
- 无法提供最小验证和回滚方式。

## 4. 高风险变更授权

除依赖外，下列变化也需要额外授权：

| 类别 | 授权前必须明确 |
|---|---|
| 数据库 | 迁移、数据量、锁、回滚、备份和触发器 |
| 鉴权与权限 | 信任边界、会话、密钥、伪造和失效路径 |
| 公共 API 和 Schema | 兼容性、调用方、错误码和版本策略 |
| Xcode 与签名 | Target、Scheme、Bundle ID、证书和环境 |
| CI 与发布 | 触发条件、权限、环境、失败停止和回滚 |
| 生产配置 | 目标环境、密钥来源、审批和审计 |
| 许可与法律 | 权利来源、适用范围和维护者书面决定 |

授权记录必须绑定任务、基线、具体文件或操作和有效期。对一个文件的授权不能自动扩大为整个目录或以后所有任务。

## 5. 依赖文件识别

项目应在任务控制清单中列出实际依赖文件，例如：

```text
package.json
package-lock.json
Podfile
Podfile.lock
pom.xml
**/pom.xml
Package.swift
Package.resolved
requirements.txt
pyproject.toml
go.mod
go.sum
```

候选边界脚本只负责判断“依赖文件被改且是否已有授权”，不解析依赖内容，也不替代许可、安全和维护性评估。

## 6. 版本和事实源

- 实际版本只在工程依赖文件和锁文件中维护；
- 白名单记录用途和批准结论，不复制形成第二套版本事实；
- 依赖升级必须同步相关架构、兼容性、测试和回滚信息；
- 依赖已批准不代表构建、运行、安全或生产验收通过。

## 7. 移除和回滚

依赖移除同样需要验证：

- 没有仍在使用的代码路径；
- 构建和锁文件已经一致；
- 缓存、生成代码和资源已正确处理；
- 运行和用户路径没有回归；
- 回滚时可以恢复原版本和配置。
