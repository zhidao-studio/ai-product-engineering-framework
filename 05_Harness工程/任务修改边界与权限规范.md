# 任务修改边界与权限规范

> 本文定义一次 AI 任务允许修改、禁止修改和需要额外授权的范围。任务 Context 说明为什么做，任务控制清单提供机器可检查的路径输入，两者必须引用同一任务和基线。

## 1. 默认规则

- 未明确授权的路径默认禁止修改；
- 禁止范围优先于允许范围；
- 高风险范围即使位于允许路径内，也必须单独批准；
- 实际修改超过任务边界时先停止，更新任务并重新批准，不得先改后补；
- 删除、移动和重命名同时视为对原路径和目标路径的影响；
- 任务结束必须列出实际修改、明确未修改区域和越界检查结果。

## 2. 三类路径

### 2.1 允许路径

任务为实现目标可以修改的最小文件或目录，例如：

```text
05_Harness工程/任务修改边界与权限规范.md
08_模板资产/Harness/
scripts/check_task_boundary.py
```

允许目录应以 `/` 结尾。允许整个仓库只能用于确实需要全仓操作的任务，并按高风险处理。

### 2.2 禁止路径

即使被允许目录覆盖也不得修改，例如：

```text
LICENSE
VERSION
生产配置/
未授权代码工程/
```

禁止路径不能通过任务内临时解释绕过；需要修改时必须建立新任务或重新批准边界。

### 2.3 高风险路径

允许修改但必须单独批准的路径或文件类型，例如：

- 依赖定义和锁文件；
- 数据库迁移、初始化和回滚脚本；
- 鉴权、权限、密钥和签名；
- Xcode 工程、Target、Scheme、签名和 Bundle ID；
- 公共 API、OpenAPI、Schema、错误码和数据格式；
- CI、部署、域名、生产配置和发布脚本；
- 根目录结构、事实源、许可证和版本文件。

项目可以增加高风险区域，但不能静默删除 Framework 的通用高风险类别。

## 3. 路径表达规则

机器控制清单支持：

- 精确文件：`README.md`；
- 目录前缀：`05_Harness工程/`；
- Glob：`**/*.sql`、`**/pom.xml`。

判断优先级：

```text
禁止路径
→ 是否在允许路径
→ 是否命中高风险路径且已批准
→ 是否命中依赖文件且已批准
→ 通过或阻止
```

## 4. 修改类型和最低要求

| 修改类型 | 最低要求 |
|---|---|
| 新增文件 | 允许新增的目录、用途和文件类型明确 |
| 修改文件 | 文件或目录在允许范围，保持约定明确 |
| 删除文件 | 说明替代来源、链接影响和回滚方式 |
| 移动或重命名 | 原路径、目标路径和引用更新均在范围内 |
| 生成文件 | 说明来源、是否应提交和再生方式 |
| 外部状态变化 | 单独记录系统、权限、影响和恢复方式 |

Git 路径检查只能判断仓库文件，不能证明数据库、云服务、设备或外部系统没有变化。外部状态必须由任务和专项证据控制。

## 5. 权限规则

任务应明确 Agent 可以使用的权限：

- 只读仓库；
- 修改工作区；
- 运行构建和测试；
- 使用网络；
- 访问设备、数据库或外部服务；
- 提交、推送或创建 PR；
- 发布、迁移或执行破坏性操作。

权限采用最小必要原则。能够修改文件不代表可以推送，能够连接数据库不代表可以迁移，能够构建不代表可以发布。

## 6. 越界处理

发现越界时：

1. 停止继续修改；
2. 保留当前差异和检测输出；
3. 判断是任务边界遗漏还是实现方向错误；
4. 可以用范围内方案完成时，撤销越界部分并继续；
5. 必须扩大范围时，更新任务风险、允许路径、验证和人工批准；
6. 涉及高风险或严重风险时建立独立任务或授权记录。

禁止通过把整个仓库加入允许路径来消除越界失败。

## 7. 人工检查与机器检查边界

机器检查适合判断：

- 文件是否在允许或禁止路径；
- 是否触碰已列出的高风险和依赖文件；
- 清单必填字段是否存在。

机器检查不能判断：

- 代码是否真的符合产品目标；
- 修改是否绕过了业务规则；
- 文件内部是否包含隐性权限或安全变化；
- 外部系统是否产生副作用；
- 高保真和实现是否视觉一致。

这些内容必须由专项检查、运行验证和人工验收补充。
