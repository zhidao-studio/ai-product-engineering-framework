# DEC-008：采用稳定版本与建设阶段双轨管理

- 状态：已接受
- 日期：2026-07-12
- 影响版本：v0.1.1 及后续版本

## 1. 背景

v0.1 完成通盘复核后，仓库已经开始建设 v0.2-A Context 可执行化，但 README、AGENTS.md 和项目 Context 中的稳定版本仍为 v0.1.0，CHANGELOG 又把 v0.1 修订和 v0.2 新能力共同放在“未发布”中。

这导致三个不同概念被混用：

1. 已经可以对外确认的稳定版本；
2. 当前正在推进的建设阶段；
3. Context Pack 自身的快照版本。

如果不分离，维护者和 AI 会误判哪些能力已经稳定、哪些仍在验证，也无法准确决定何时增加修订版本或次版本。

## 2. 决定

采用三类独立编号：

```text
稳定发布版本：v0.1.1
目标发布版本：v0.2.0
当前建设阶段：v0.2-A Context 可执行化
Context Pack 版本：0.2-A.x
```

并新增根目录 `VERSION` 作为当前稳定发布版本的单一事实源。

## 3. 版本规则

- v0.1.1 用于承载 v0.1 范围内的全仓 Review、事实源收敛、生命周期口径统一、链接图片修复和版本治理；
- v0.2-A 的 Context 规范、候选模板和自应用验证仍属于目标版本 v0.2.0 的开发内容，不因存在于 `main` 就视为稳定发布；
- 只有 v0.2 Roadmap 退出标准满足后，才能发布 v0.2.0；
- Context Pack 使用独立字段，不再使用含义不清的通用 `version` 同时表达稳定版本和 Pack 版本。

## 4. 权威顺序

```text
VERSION
→ CHANGELOG
→ Roadmap
→ 版本管理规范
→ README / AGENTS / 项目 Context 摘要
```

发生版本冲突时，不得静默选择，必须优先修正摘要文件。

## 5. 影响

### 正向影响

- 明确区分已发布能力与开发中候选能力；
- 修订版本可以准确记录 Review 和治理修正；
- Roadmap 阶段不再被误认为发布版本；
- Context Pack 可以独立演进而不污染框架版本；
- 后续自动门禁可以直接检查版本一致性。

### 成本与风险

- README、AGENTS、CHANGELOG、Roadmap 和项目 Context 需要同步维护；
- 在自动门禁建立前，版本一致性仍依赖人工检查；
- `main` 分支同时包含稳定基线和下一版本候选内容时，必须清楚标注成熟度。

## 6. 后续动作

- 将当前稳定版本更新为 v0.1.1；
- 将 v0.1 Review 修订从 CHANGELOG“未发布”移动到 v0.1.1；
- 保留 Context 可执行化在“未发布”；
- 更新 README、AGENTS、Roadmap 和 Framework 项目 Context；
- 新增 v0.1.1 全仓复核报告；
- 在 Harness 阶段增加版本一致性检查门禁。
