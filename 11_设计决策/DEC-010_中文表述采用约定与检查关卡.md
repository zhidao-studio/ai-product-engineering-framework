# DEC-010：中文表述采用“约定”与“检查关卡”

## 1. 背景

Framework 早期使用“契约”和“门禁”表达 Contract 与 Gate。术语在工程领域成立，但对非技术产品负责人、个人开发者和跨职能参与者不够直观，也容易被误解为法律合同或访问控制系统。

Framework 的目标之一是让非技术人员也能判断项目是否正确推进，因此中文表达需要在不损失工程含义的前提下降低理解成本。

## 2. 决策

自 v0.1.4 起：

- 面向中文读者的主要表述使用“约定”；
- 面向中文读者的主要表述使用“检查关卡”；
- 首次需要对照专业概念时，可以写“约定（Contract）”“检查关卡（Gate）”；
- `OpenAPI`、`Schema`、`Contract`、`Gate`、`GATE-xxx` 和机器字段保持原技术名称；
- 已发布历史报告和验证记录不追溯重写，按本决策解释旧称；
- 当前有效规范、导航、模板说明和项目落地指导必须采用新表述。

详细规则由：[术语与易懂表达规范](../01_框架定义/术语与易懂表达规范.md) 定义。

## 3. 不改变的内容

本决策不改变：

- 系统之间必须共同遵守且可检查的规则含义；
- 未通过检查时不得进入下一阶段的控制含义；
- Harness 的六类控制面；
- 十阶段产品价值生命周期；
- Context、Harness、Skills、Agents、Loop 的边界；
- 机器标识、接口标准和外部协议名称。

## 4. 影响

- 当前中文文档更容易被产品、设计、运营和非专业开发者理解；
- YouYu 等参考工程可以直接使用中文目录“接口与数据约定”“执行控制与检查关卡”；
- 历史资料和当前资料可能短期存在新旧称并存，需要通过术语规范解释；
- 后续自动检查应识别当前有效文档中的旧称，并允许历史目录中的旧称。

## 5. 验证

- 当前权威文档完成术语替换；
- Harness 权威文档迁移到易懂文件名；
- README、AGENTS、Roadmap、Context 和参考工程说明一致；
- 保留技术名和机器编号未被误改；
- 新增 v0.1.4 发布报告和任务记录。
