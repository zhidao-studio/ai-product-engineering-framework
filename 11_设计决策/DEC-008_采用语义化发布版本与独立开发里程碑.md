# DEC-008：采用语义化发布版本与独立开发里程碑

- 状态：已接受
- 日期：2026-07-12
- 影响版本：v0.1.1 及后续版本

## 1. 背景

仓库在 v0.1.0 完成框架立项后，又进行了全量 Review、事实源收敛和 Context 可执行化建设。期间出现了三类编号同时使用：

- `v0.1.0`：框架稳定发布版本；
- `v0.2-A`：v0.2 开发阶段的内部里程碑；
- `v0.2-A.1`：Framework 项目 Context Pack 的内部版本。

如果不区分这些编号，容易出现：

- 将 Review 修订继续写成 v0.1，而没有形成可追溯的小版本；
- 将开发里程碑误认为已经发布的版本；
- 将 Context Pack 版本误认为框架发布版本；
- CHANGELOG 的未发布内容同时混入旧版本修订和下一版本能力。

## 2. 决定

采用以下版本体系：

1. 框架发布版本使用 `v主版本.次版本.修订版本`；
2. 既有版本范围内的 Review、错误修复和一致性治理增加修订版本；
3. 新增一组可独立验证的能力增加次版本；
4. `A/B/C` 仅作为次版本开发过程中的内部里程碑；
5. Context Pack 使用 `context_pack_version` 独立编号；
6. 稳定版本、目标版本和 Pack 版本必须使用不同字段。

当前基线：

```text
v0.1.0：框架立项初版
v0.1.1：全量 Review 与治理修订
v0.2.0：Context 与 Harness 可执行化，开发中
```

## 3. v0.1.1 的范围

v0.1.1 只包含对 v0.1.0 既有范围的修订：

- 全仓库通盘复核；
- 删除平行事实源；
- 统一十阶段生命周期口径；
- 修复图片和内部导航；
- 修正过期 README 结构规范；
- 建立版本管理规范和贡献规范；
- 明确稳定版本与开发里程碑的区别。

Context 可执行化规范、候选模板和自应用验证属于 v0.2.0 的未发布开发内容，不因仓库稳定版本升级到 v0.1.1 而自动变为稳定能力。

## 4. 影响

### 正向影响

- Review 修订具有明确版本和变更记录；
- README、AGENTS、Roadmap、CHANGELOG 和 Context 可以表达相同版本状态；
- 开发中的候选能力不会被误认为稳定发布；
- 后续自动门禁可以校验明确字段，而不是解析含义不清的 `version`。

### 成本与风险

- 需要同步修改多个现有文档和模板；
- `main` 分支可能同时包含稳定基线和下一版本候选内容，必须持续明确状态；
- 当前尚无自动版本一致性检查，仍需人工 Review。

## 5. 后续动作

- 新增《版本管理规范》；
- 将 CHANGELOG 拆分为 v0.1.1 和未发布 v0.2.0；
- 更新 README、AGENTS、Roadmap 和 Framework 项目 Context；
- 将 Context Pack 的 `version` 字段改为 `context_pack_version`；
- 统一机器状态枚举与中文展示名称；
- 在 Harness 阶段增加版本与状态一致性门禁。

## 6. 替代条件

只有在真实发布流程证明当前三段式版本无法表达框架演进，或仓库正式采用另一套发布系统时，才可以通过新的设计决策替代本规则。不得在单个文档中静默修改版本语义。
