# DEC-003：区分产品生命周期与 AI 工程基础设施

- 状态：已接受
- 日期：2026-07-12
- 决策人：项目负责人
- 关联文档：`02_全局模型/AI产品工程全局框架.md`

## 背景

早期方案曾将战略、产品、设计、高保真、工程、Context、Agent、执行、验证和反馈全部排列为一条十层或十一层流程。

## 问题

Context、Harness、Skills、Agents 和 Loop 实际贯穿多个生命周期阶段。如果把它们作为顺序阶段，会产生错误理解，例如 Context 只在工程规格后出现，Loop 只在发布后发生。

## 备选方案

1. 继续使用单一多层模型；
2. 使用主流程加附属能力，但不明确治理；
3. 区分产品交付生命周期、AI 工程基础设施、治理与组织记忆三个平面。

## 决定

采用方案 3。

## 原因

- 明确“顺序阶段”和“横向能力”的差异；
- 支持每个阶段使用 Context、Harness 和 Loop；
- 方便后续 Skills、门禁和平台适配找到稳定位置；
- 将决策、安全、成本和变更从流程步骤提升为全局治理。

## 影响

README 和核心文档统一使用三平面模型。后续不得未经设计决策重新将所有概念扁平化成单一路径。

## 验证

新增能力能够判断它属于生命周期阶段、横向基础设施或治理能力；框架图中不再出现概念顺序误导。
