# DEC-009：采用专有闭源许可并限制未经授权使用

- 状态：已接受
- 日期：2026-07-12
- 影响版本：v0.1.2 及后续版本
- 决策责任人：zhidao-studio

## 1. 背景

仓库此前未提交正式 `LICENSE`，部分文档使用“开放框架”“开放标准”等表述，容易被理解为允许外部复制、修改、分发或商业使用。

项目维护者已经明确决定：

> 本仓库采用闭源、专有权利模式，不允许未经授权的个人或组织使用。

GitHub 官方文档说明，没有许可证时默认版权法适用，权利人保留全部权利；但公开仓库仍允许 GitHub 用户按照平台服务条款查看和 Fork。因此，仅添加专有许可不能阻止公开可见性，若要限制访问，还必须将仓库可见性改为 Private。

## 2. 决定

1. 在根目录新增 `LICENSE`，采用专有许可和“保留全部权利”声明；
2. 明确本项目不是开源项目，不授予复制、使用、修改、分发、商业化、模型训练或创建衍生作品的权限；
3. 任何授权必须由 zhidao-studio 单独书面作出；
4. 未经邀请的外部贡献不被接受；
5. 将“开放框架”“开放标准”调整为“专有维护、跨平台、可验证的框架”；
6. 稳定修订版本由 v0.1.1 升级为 v0.1.2；
7. 将仓库改为 Private 作为必须由维护者在 GitHub 设置中执行的治理动作。

## 3. 权利边界

除 GitHub 平台条款为浏览公开仓库所允许的有限平台行为外，未经书面授权，不允许：

- 使用或实施框架内容；
- 复制、下载、镜像或转载；
- 修改、翻译或创建衍生作品；
- 分发、再许可、出售或商业交付；
- 纳入其他产品、服务、课程或内部工程体系；
- 用于模型训练、微调、评测或数据集建设。

正式条款以根目录 `LICENSE` 为准。

## 4. 公开仓库风险

当前仓库仍为 Public。其影响包括：

- 内容可以被外部用户看到；
- GitHub 平台条款允许用户查看和 Fork；
- 已经存在的 Fork 或本地副本不会因为后续改为 Private 自动消失；
- 专有许可提供法律边界，但不能替代访问控制。

因此，若目标是“不允许其他人访问和使用”，维护者必须尽快将仓库设为 Private，并检查既有 Fork、协作者和外部副本风险。

## 5. 影响

### 正向影响

- 明确仓库不是开源项目；
- 消除外部使用权的歧义；
- 为内部协作、授权和商业化保留全部权利；
- 为后续文档和 Agent 行为提供统一法律边界。

### 成本与风险

- Public 状态下仍无法阻止查看和 Fork；
- 自定义专有许可的具体法律效力可能因司法辖区和事实情况不同；
- 重要商业使用、授权或维权场景应由专业律师审核；
- 历史上已被复制的内容无法仅通过新增许可自动收回。

## 6. 后续动作

- 更新 README、AGENTS、CONTRIBUTING、愿景定位和边界声明；
- 更新 VERSION、CHANGELOG、Roadmap、版本规范和项目 Context；
- 将仓库可见性切换为 Private；
- 审查协作者权限、Fork 和敏感信息；
- 在 Harness 阶段增加许可证、版权声明和仓库可见性检查。

## 7. 替代条件

只有项目维护者明确决定开放源代码、商业授权或采用其他许可模式时，才可通过新的设计决策替代本决策。任何 Agent、贡献者或局部文档不得自行更换许可证。
