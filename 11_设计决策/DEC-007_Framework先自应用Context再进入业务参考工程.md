# DEC-007：Framework 先自应用 Context，再进入业务参考工程

- 状态：已接受
- 日期：2026-07-12
- 影响版本：v0.2-A

## 背景

Context 规范和候选模板已经建立，但仅凭文档完整不能证明其可执行。直接在复杂业务工程中使用，可能同时暴露框架问题和业务问题，难以判断根因。

Framework 仓库自身存在跨会话、跨模型、长期决策和多文档一致性需求，适合作为第一轮受控自应用场景。

## 决定

v0.2-A 采用两步验证：

1. **Framework 自应用验证**：为本仓库建立项目 Context Pack、阶段 Context、任务 Context Pack 和完整性检查，验证能否从仓库恢复项目认知并保持单一事实源；
2. **业务参考工程验证**：在 H5、移动端、后端、数据或 AI 内容产品中选择一个真实任务，验证 Context 能否支持产品与代码交付。

Framework 自应用通过不等于 Context 模板已经稳定，也不满足 v0.2-A 的全部退出标准。

## 自应用验证重点

- 新会话能否理解 Framework 是什么、不是什么；
- 能否恢复十阶段生命周期和五大基础设施；
- 能否识别当前建设阶段与下一步；
- 能否定位宪法、全局模型和设计决策的权威来源；
- 能否发现重复事实源和口径冲突；
- 能否使用阶段和任务 Context 控制仓库修改；
- 任务结束后是否能够回写 Roadmap、CHANGELOG 和经验。

## 约束

- 自应用 Context 放在独立目录，不复制核心文档正文；
- Pack 以链接、状态和索引为主；
- 发现 Framework 自身不符合 Context 规范时，必须先修正事实源；
- 候选模板最多标记为“自应用初步验证”，不能标记为稳定；
- 完成自应用后必须选择外部或业务参考工程继续验证。

## 影响

新增 `12_框架项目Context`，用于保存本仓库自身的项目、阶段、任务和检查记录。该目录是 Framework 的运行 Context，不是新的核心模型目录。
