# DEC-004：设立框架宪法层

- 状态：已接受
- 日期：2026-07-12
- 影响版本：v0.1.0 及后续版本

## 背景

框架已经形成产品交付生命周期、AI 工程基础设施和治理与组织记忆三个平面，并持续增加 Context、Harness、Skills、Agents、Loop、角色、模板和参考工程等内容。

如果只继续增加模块而缺少稳定的上位判断标准，项目容易出现以下问题：

- 随任务重点变化反复调整核心模型；
- 新增能力各自合理，但整体方向不一致；
- 框架退化为 Prompt、Skill、模板或工具集合；
- 不同 AI 会话无法恢复关键原则和设计边界；
- 自动化、效率和功能数量压过用户价值、责任与验证。

## 决定

在 `01_框架定义` 中设立“框架宪法层”，由以下四类事实源共同组成：

1. 《AI 产品工程框架：愿景与定位》；
2. 《适用场景与期望》；
3. 《AI 产品工程核心原则》；
4. 《AI 产品工程边界声明》。

宪法层负责定义：

- 框架为什么存在；
- 解决什么问题和服务哪些场景；
- 哪些原则不可被局部实现破坏；
- 框架负责什么、不负责什么；
- 人和 AI 的最终责任如何划分；
- 核心模型和边界如何变更。

## 约束

- Context、Harness、Skills、Agents、Loop、模板、门禁、平台适配和参考工程必须能够说明其与宪法层的关系。
- 新能力不能只以“技术可行”或“社区流行”作为进入框架的理由。
- 偏离核心原则或改变边界时，必须新增设计决策，不能只修改 README。
- 宪法层不是永久不可修改，但修改必须说明原因、影响、替代关系和生效版本。

## 影响

### 正向影响

- 为后续模块细化提供统一判断标准；
- 降低跨会话、跨模型和长期演进中的认知漂移；
- 明确框架与具体 AI Coding 工具的上下层关系；
- 防止项目过早进入自动化平台和 Skill 堆积。

### 成本与风险

- 新增能力需要完成原则与边界检查；
- 宪法层可能因表达过度抽象而失去可执行性，因此必须通过参考工程持续验证；
- 如核心原则数量不断膨胀，需在后续版本合并和精简，而不是无限增加条款。

## 后续动作

- 在 README 文档导航中显式列出宪法层四份文档；
- 在 `AGENTS.md` 中保持核心模型和变更要求一致；
- 后续模板和 Skill 增加“宪法检查”字段；
- 使用真实参考工程验证核心原则是否可执行。
