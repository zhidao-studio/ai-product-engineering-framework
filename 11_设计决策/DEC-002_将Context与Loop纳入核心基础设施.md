# DEC-002：将 Context 与 Loop 纳入核心基础设施

- 状态：已接受
- 日期：2026-07-12
- 决策人：项目负责人
- 关联文档：`04_Context工程/Context与记忆管理.md`、`07_Loop工程/持续反馈与演进闭环.md`

## 背景

随着对话时间和任务重点变化，模型可能忘记早期结论。与此同时，一次性“生成结果”无法解决运行后的用户反馈、任务失败和框架自身改进。

## 问题

如何让 AI 团队获得稳定的组织记忆，并将执行和产品结果持续带回下一轮？

## 备选方案

1. 依赖模型长期记忆；
2. 只使用 `AGENTS.md` 或 `CLAUDE.md`；
3. 建立分层 Context、设计决策和多级 Loop。

## 决定

采用方案 3。Context 与 Loop 与 Harness、Skills、Agents 并列，成为框架五大 AI 工程基础设施。

## 原因

- 会话记忆不是可靠的项目事实来源；
- 单个指令文件无法承载完整产品、设计、工程和历史决策；
- 反馈必须有归因、停止条件和长期沉淀位置；
- Framework 本身也需要通过参考工程反馈持续改进。

## 影响

仓库必须维护 Context 规则、设计决策目录和 Loop 模型。重要结论不得只存在于聊天中。

## 验证

更换模型、工具或新开会话后，参与者能够通过仓库恢复项目核心认知；参考工程中的失败能够明确更新到 Context、Harness、Skill、Agent 契约或产品定义。
