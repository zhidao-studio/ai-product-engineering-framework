# 模板资产

本目录存放把 Framework 标准转化为可执行输入和输出的模板。

模板不是越多越好。正式模板必须来自真实任务，证明能够减少遗漏、歧义或风险，并有明确适用边界。

中文术语遵循：[术语与易懂表达规范](../01_框架定义/术语与易懂表达规范.md)。

## 1. 模板族

| 生命周期或基础设施 | 计划或已有模板 | 状态 |
|---|---|---|
| 价值验证 | 产品机会评估、用户问题验证、成功指标 | 计划 |
| 产品定义 | PRD、不做清单、业务规则、验收断言 | 计划 |
| 体验与高保真 | 用户流程、页面清单、状态矩阵、高保真确认记录 | 计划 |
| [工程规格](工程规格/数据库设计基础规范.md) | 架构、API、数据库、前端、后端、移动端规格 | 数据库规范单项目已验证 |
| [Context](Context/README.md) | 项目 Pack、阶段 Context、任务 Pack、冲突和经验回写 | 项目与任务模板单项目已验证；模板族整体仍为候选 |
| Harness | 修改边界、阶段检查关卡、约定一致性检查、发布前检查 | B1 风险与阶段基线为候选，模板待 B2 |
| Skills 与 Agents | Skill 定义、角色输入输出约定、编排清单 | 计划 |
| 验证 | 代码审查、运行验证、模拟用户验收、验证报告 | 计划 |
| Loop | 反馈分类、指标复盘、失败归因、改进记录 | 计划 |

## 4. 模板准入规则

模板进入正式目录前必须：

1. 在至少一个真实参考工程中使用；
2. 证明能够减少遗漏、歧义或风险；
3. 明确必填项、可裁剪项和禁止删除项；
4. 有清晰正例、反例和检查清单；
5. 能被人和 AI 共同理解；
6. 说明适用范围、风险等级和已知限制；
7. 不复制形成第二套权威事实源。

## 5. 模板成熟度

```text
候选
→ 单项目已验证
→ 跨项目已验证
→ 稳定
→ 已废弃
```

状态变化必须有验证证据；结构性变化需要同步规范，必要时形成设计决策。

## 6. 当前约束

- 不批量生成未经真实项目验证的空模板；
- 不因为社区流行就直接纳入稳定资产；
- 不允许模板替代产品判断和人工批准；
- 不把模板复杂度当作工程成熟度；
- 小任务可以裁剪，但不得删除目标、边界、验证和责任。
