# 参考工程

> 参考工程用于证明 Framework 能够控制真实产品交付，而不是只形成漂亮文档。当前运行状态以 [Framework 项目 Context](../12_框架项目Context/README.md) 为准。

中文术语遵循：[术语与易懂表达规范](../01_框架定义/术语与易懂表达规范.md)。

## 1. 参考工程必须覆盖

- 价值和产品范围；
- 用户流程与高保真预览；
- 工程架构、API 和数据约定；
- 项目 Context 和任务上下文包；
- 至少一项受控 AI 实现任务；
- 静态审查、运行验证和模拟用户验收；
- 发布或可运行交付；
- 用户反馈或代理指标；
- 对 Framework 的反向改进记录。

## 2. 候选类型

- H5 小型产品：验证前端、交互、高保真和轻量检查关卡；
- iOS 应用：验证移动端规范、状态、设备体验和发布流程；
- Go 或 Java 后端：验证 API、数据、依赖和服务端检查关卡；
- 数据应用：验证数据口径、Schema、批处理和质量检查；
- AI 内容产品：验证生成、人工质检、发布指标和内容 Loop。

## 4. 正式验证启动条件

正式业务参考任务至少需要：

1. 完成三个代码工程的依赖恢复、构建、启动和基础运行复核；
2. 关闭或明确处置安全、配置和鉴权信任边界 P0/P1；
3. 基于通过复核的 `main` 建立正式项目 Context Pack；
4. 产品范围、用户流程和高保真版本经过人工确认；
5. 前后端接口约定和数据结构约定有唯一权威来源；
6. 建立阶段 Context 和正式任务 Context Pack；
7. 配置修改边界、检查关卡和人工确认节点；
8. 选择一个小而完整的前后端业务切片；
9. 完成静态、运行和模拟用户三层验证；
10. 记录填写成本、人工修正、遗漏和 Framework 回写。

在第 1、2 项完成并经人工确认前，不得将正式业务任务状态设置为 `ready` 或 `in_progress`。

## 5. 通过标准

参考工程不能仅展示最终代码。必须能够从任一产物追溯到：

```text
产品目标 → 设计与高保真 → 工程约定 → 任务包 → 修改 → 验证 → 反馈 → 框架改进
```

一次业务任务只有在以下证据完整后，才能作为 `single_project_validated` 的依据：

1. 上游产品、体验和工程事实可追溯且已批准；
2. 项目、阶段和任务 Context 能被新 Agent 独立恢复；
3. 修改边界、约定和检查关卡真实生效；
4. 静态、运行和模拟用户三层验证全部完成；
5. 结果、失败、人工修正和填写成本已经记录；
6. 长期事实源和 Framework 改进项完成回写；
7. 由明确的人类责任人批准验证结论。

## 6. 与实战仓库的关系

`ai-product-engineering-in-action` 可以承担教学与步骤演示；本仓库的参考工程入口负责记录对 Framework 的正式验证结论、缺陷和改进要求。
