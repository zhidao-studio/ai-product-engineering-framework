# 参考工程

参考工程用于证明 Framework 能够控制真实产品交付，而不是只形成漂亮文档。

## 1. 参考工程必须覆盖

- 价值和产品范围；
- 用户流程与高保真预览；
- 工程架构、API 和数据契约；
- 项目 Context 和任务上下文包；
- 至少一项受控 AI 实现任务；
- 静态审查、运行验证和模拟用户验收；
- 发布或可运行交付；
- 用户反馈或代理指标；
- 对 Framework 的反向改进记录。

## 2. 候选类型

- H5 小型产品：验证前端、交互、高保真和轻量门禁；
- iOS 应用：验证移动端规范、状态、设备体验和发布流程；
- Go 或 Java 后端：验证 API、数据、依赖和服务端门禁；
- 数据应用：验证数据口径、Schema、批处理和质量检查；
- AI 内容产品：验证生成、人工质检、发布指标和内容 Loop。

## 3. 通过标准

参考工程不能仅展示最终代码。必须能够从任一产物追溯到：

```text
产品目标 → 设计与高保真 → 工程契约 → 任务包 → 修改 → 验证 → 反馈 → 框架改进
```

## 4. 与实战仓库的关系

`ai-product-engineering-in-action` 可以承担教学与步骤演示；本仓库的参考工程入口负责记录对 Framework 的验证结论、缺陷和改进要求。
