## 1. 真实观察

YouYu 账号与“我”不是一次生成完成。项目经历了高保真候选修订、维护者批准、功能与视觉未对齐返工、分支与主线分叉、服务端真实运行、模拟器严格路径、真机多轮反馈、协议交互与对齐修正，最终由维护者确认核心路径并合并 PR #3。

## 2. 已被一个真实项目证明的做法

### 2.1 高保真与功能必须使用同一集成基线

“代码能构建”不能证明页面符合批准高保真。受控实现开始前必须固定高保真版本；完成时必须逐页对照，视觉层和功能层属于同一交付单元。

### 2.2 构建、安装、启动、自动化和人工验收必须分开记录

```text
构建通过
≠ 安装成功
≠ 进程启动
≠ 自动化主路径通过
≠ 用户接受
```

YouYu 的最终通过来自真机上的维护者操作确认，不是从构建日志推断。

### 2.3 任务、分支、PR 和验收需要一条追溯链

关闭未合并的旧分支不能作为交付。TASK-013 在最新主线重新集成，记录工作分支、验证结果、人工批准、PR 和合并提交，才形成可追溯交付。

### 2.4 Framework 同步应服从参考工程运行

Framework 曾因持续静态 Review 和版本同步增加成本。冻结后把唯一动作改为跑通 YouYu，从第一个真实运行错误开始修复，最终取得可接受产品。Framework 应在业务闭环后一次性回写，而不是每个静态发现都发布一个框架版本。

### 2.5 外部平台阻塞必须显式隔离

GitHub Actions 因 Billing 或额度未执行时，应记录为外部证据缺口；在维护者批准下，可以用等价本地命令和真机证据继续开发，但不得把未运行的远程检查写成通过。

## 3. 发现的 Context 缺口

- 项目 Context 的提交字段需要频繁同步，维护成本高；
- 阶段 Context 在项目进入执行和验收后仍停留在业务准备，缺少更新触发；
- 经验记录已形成，但“候选→采纳→复验”的流程尚未跑完；
- Context 成本、人工修正次数和遗漏没有结构化汇总。

## 3.1 本轮修订结果

- YouYu EXP-005 记录了 2026-07-18 至 2026-07-22 的可核验成本：项目 Context 23 次提交、阶段目录 0 次提交、至少 8 个明确修正事件；
- Token、人工工时、模型与工具费用因无可信统一事实源标为 `not_measured`；
- “业务功能准备阶段”已归档，当前入口切换到“质量与安全验证阶段”；
- 阶段模板增加来源提交、最近确认、替代关系、复核触发和过期条件；
- CTX-CHECK-002 已完成当前阶段复验，但下一次真实阶段转换尚未验证。

## 4. 适用边界

本反馈只经过 YouYu 一个 SwiftUI + Spring Boot 业务切片验证，可以指导相似任务，但不能升级为跨项目或稳定规则。下一次应在另一类业务切片或另一参考工程复验。
