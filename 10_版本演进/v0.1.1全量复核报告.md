# v0.1.1 全量复核报告

> **历史快照：** 本报告记录 v0.1.1 全量复核时的仓库状态。文件中的稳定版本、许可证待决项、术语和下一步只反映当时判断；当前稳定版本、工作段、阻塞和下一步以 [Framework 项目 Context](../12_框架项目Context/README.md)、[CHANGELOG](../CHANGELOG.md) 和 [Roadmap](Roadmap.md) 为准。

- 复核日期：2026-07-12
- 复核版本：v0.1.1
- 目标开发版本：v0.2.0
- 当前开发里程碑：A / Context 可执行化
- 复核范围：根目录规范文件、框架宪法、全局模型、角色体系、Context、Harness、Skills 与 Agent、Loop、模板、参考工程、Roadmap、设计决策、Framework 自身 Context 和图片资产
- 核心模型变化：无
- 复核结论：**v0.1.1 可以作为 v0.1.0 的稳定修订基线；v0.2.0 的 Context 能力仍处于候选和业务验证前状态。**

## 1. 总体判断

当前仓库已经形成完整且可解释的 AI 产品工程顶层框架：

1. 产品价值生命周期描述产品如何从价值验证走向持续反馈；
2. Context、Harness、Skills、Agents、Loop 作为横向基础设施贯穿生命周期；
3. 治理与组织记忆管理决策、版本、安全、权限、质量、指标、成本、变更和回滚；
4. 高保真原型预览、模拟用户验收、发布交付和持续反馈都有独立位置；
5. README、AGENTS、专题文档、设计决策、项目 Context 和 CHANGELOG 已形成分层事实体系。

本轮没有发现需要推翻宪法、三平面模型、十阶段生命周期或五大基础设施的证据。

## 2. 已确认的优势

### 2.1 顶层模型边界清晰

生命周期、AI 工程基础设施和治理没有被混成一条无限增长的流水线。Context、Harness 和 Loop 被正确视为横切能力。

### 2.2 人类最终责任明确

价值判断、高保真确认、高风险权限、验收豁免、发布和回滚仍由人类角色承担。AI 的自我判断不能替代证据和批准。

### 2.3 产品体验没有被代码实现替代

高保真预览是编码前的正式门禁；模拟用户验收与发布交付相互独立，避免“代码完成等于产品完成”。

> “门禁”是本历史报告形成时使用的原始术语。当前中文主要表述使用“检查关卡”。

### 2.4 Context 已从理念进入可执行设计

已经具备：

- 框架、项目、阶段、任务四级作用域；
- 权威事实源和状态管理；
- 项目、阶段和任务 Context Pack；
- 装配、裁剪、冲突、过期和敏感信息处理；
- 决策、验证和经验回写；
- 完整性清单和候选模板；
- Framework 自应用检查。

### 2.5 复用顺序成熟

仓库明确采用：

```text
真实问题 → 实际执行 → 结果验证 → 经验归纳 → 候选资产 → 跨项目复验 → 稳定能力
```

没有把社区流行、模型建议或一次成功直接写成稳定 Skill。

## 3. 本轮发现和修复

### P0-1：版本边界混写

**发现：** `CHANGELOG.md` 的“未发布”同时包含 v0.1 Review 修订和 v0.2 Context 新能力；README 和项目 Context 仍标记 v0.1.0。

**风险：** 无法判断哪些能力已经稳定，哪些仍在开发；开发里程碑可能被误认为发布版本。

**修复：**

- 新增《版本管理规范》；
- 新增 DEC-008；
- 稳定版本升级为 v0.1.1；
- 目标开发版本明确为 v0.2.0；
- `A/B/C` 明确为开发里程碑，不是发布版本；
- CHANGELOG 拆分 v0.1.1 和未发布 v0.2.0。

### P0-2：README 结构规范包含已删除目录

**发现：** `README结构规范.md` 仍推荐 `02_核心模型`、`05_设计决策记录` 等旧目录。

**风险：** 后续 AI 会根据规范重新创建已经删除的平行事实源。

**修复：** 全量重写 README 结构规范，与当前十二个权威模块和版本状态一致，并明确禁止恢复旧目录。

### P1-1：Context 状态枚举不统一

**发现：** YAML 使用 `active`、`ready`、`verifying`，正文和状态图使用“当前有效”“进行中”“验证中”，但没有正式映射。

**风险：** 人工阅读尚可，自动门禁无法稳定判断；`active` 在事实源和阶段中含义不同但未解释。

**修复：** 建立事实源、阶段、任务和成熟度四组统一机器枚举及中文显示映射。YAML 和脚本使用英文机器值，正文可以显示中文。

### P1-2：Context Pack 版本字段歧义

**发现：** `version`、`project_context_version` 等字段可能同时指发布版本和 Pack 版本。

**风险：** v0.2-A.1 容易被误认为框架发布版本。

**修复：** 统一使用：

- `context_pack_version`；
- `project_context_pack_version`；
- `stage_context_pack_version`；
- `task_context_pack_version`；
- `stable_release`；
- `target_release`；
- `source_version`；
- `source_commit`。

### P1-3：任务规范示例阶段错误

**发现：** 登录页面代码修改任务被标记为“用户体验设计”，但实际已经进入开发执行。

**风险：** 生命周期阶段、角色责任和验证门禁错位。

**修复：** 将示例改为“受控任务执行”，并补充实现类任务的阶段映射规则。

### P1-4：Framework 自应用状态没有闭环

**发现：** 检查已经完成，但项目 Context、阶段 Context 和任务 Pack 仍写“进行中”“等待项目维护者确认”。

**风险：** 当前状态与事实不一致，违反 Context 回写原则。

**修复：**

- TASK-20260712-001 改为 `completed`；
- Framework 自应用标记为已通过并人工接受；
- 阶段 Context 勾选自应用门禁；
- 项目 Context 更新稳定版本、目标版本和下一步；
- CTX-CHECK-001 完成回写。

### P1-5：缺少贡献规范

**发现：** README 结构规范要求贡献入口，但仓库没有 `CONTRIBUTING.md`。

**修复：** 新增中文贡献指南，定义贡献分类、事实源、验证、版本、图示和敏感信息要求。

### P1-6：开源许可证尚未决定

**发现：** 仓库公开，但不存在 `LICENSE` 文件。

**处理：** 未擅自选择许可证。在 CONTRIBUTING、项目 Context 和本报告中列为项目维护者待决事项。

> 该问题已在 v0.1.2 通过专有 LICENSE 和 DEC-009 解决。

## 6. 仍需保留的风险

### P1：业务参考工程尚未验证

Framework 自应用只能证明仓库记忆和一致性管理，不能证明产品、设计、代码、企业权限和多平台场景。

### P1：自动门禁尚未建立

仍需建设：

- 内部链接检查；
- Mermaid 基础语法检查；
- 重复目录和同义事实源检查；
- 版本字段和状态枚举检查；
- Context 元数据完整性检查；
- 已替代文档引用检查。

### P1：许可证待决定

许可证必须由维护者确认，不能由 Agent 自动选择。

> 该风险已在 v0.1.2 解决。

### P2：状态摘要存在重复维护

README、AGENTS、Roadmap 和项目 Context 都需要描述当前状态。后续应通过机器元数据或自动生成减少漂移。

> v0.1.5 已将详细当前状态收敛到 Framework 项目 Context，但自动生成和一致性检查仍待 Harness 里程碑 B。

### P2：平台适配尚未验证

Claude Code、Codex、Kimi、GLM 的项目指令、记忆、Skill 和工具能力不同，目前只保持核心标准平台无关。

## 8. 下一步

按依赖顺序推进：

1. 选择一个小而完整的真实业务参考工程任务；
2. 使用项目、阶段和任务 Context Pack 完成真实任务；
3. 在真实任务中交叉建设必要的 Harness 门禁；
4. 记录填写成本、人工修正、冲突、遗漏、任务成功率和验证证据；
5. 决定 Context 模板是否达到 `single_project_validated`；
6. 再进入 v0.2.0 / B 的系统化 Harness 可执行化；
7. 由项目维护者选择开源许可证。

> 上述是 v0.1.1 复核结束时的历史计划。当前第一步是等待 YouYu 基础框架整理完成并复核整理后的 `main`；许可证已经改为专有许可。当前状态详见 [Framework 项目 Context](../12_框架项目Context/README.md)。
