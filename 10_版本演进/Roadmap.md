# Roadmap

> 路线图按“先验证方法，再封装能力，最后自动编排”的顺序推进。版本号表示框架成熟度，不表示必须一次完成全部自动化。

## v0.1：框架立项（已完成）

目标：建立完整地图和统一语言。

已完成：

- 愿景、定位、边界和适用场景；
- 框架宪法和三平面全局模型；
- 十阶段产品价值生命周期；
- Context、Harness、Skills、Agents、Loop；
- 人类与 AI 角色；
- 设计决策、模板和参考工程入口；
- Mermaid、SVG 与 ChatGPT 生成的真实图片资产；
- v0.1 通盘复核和事实源收敛。

退出标准：新参与者能够理解框架结构，后续新增内容都有明确归属，不存在已知平行权威目录。

状态：**已通过。**

复核报告：[v0.1 通盘复核报告](v0.1通盘复核报告.md)。

## v0.2：Context 与 Harness 可执行化（进行中）

目标：让一次真实 AI 任务能够获得正确 Context，在明确边界和门禁下完成并留下可验证证据。

### v0.2-A：Context 可执行化（当前阶段）

建设内容：

- 四级 Context 作用域与独立回写层；
- 权威事实源、状态、版本、新鲜度和敏感级别；
- 项目 Context Pack；
- 阶段 Context；
- 任务 Context Pack；
- Context 装配、裁剪、预算和安全规则；
- 冲突、缺失和过期处理；
- 决策、证据和经验回写；
- Context 完整性检查清单；
- 第一批候选模板。

退出标准：使用一个真实任务证明，新的人员或 Agent 不依赖历史会话也能恢复项目背景、识别权威事实、发现冲突、明确修改范围，并知道结果应回写到哪里。

当前状态：**规范和候选模板已建立，等待参考工程验证。**

### v0.2-B：Harness 可执行化

建设内容：

- 标准任务准备门禁；
- 阶段进入与退出门禁；
- 文件和模块修改边界；
- OpenAPI、Schema、依赖和行为契约门禁；
- 权限、安全、成本和停止条件；
- 三层验证模板；
- 模拟用户验收与发布门禁分离；
- 变更、发布和回滚清单；
- 第一套可运行门禁脚本。

退出标准：在参考工程中证明门禁能够发现真实问题，并且不会因为过度规则阻止正常任务。

### v0.2-C：真实任务联合验证

建设内容：

- 选择一个小而完整的参考任务；
- 使用项目、阶段和任务 Context Pack；
- 使用 Harness 门禁完成受控实现；
- 记录 Context 缺失、人工修正、任务成功率、成本和验证结果；
- 回写 Framework 缺陷和模板改进。

退出标准：Context 与 Harness 形成一次完整闭环，不只是文档规范。

## v0.3：Skills 标准化

目标：把已在真实任务中验证的方法封装为能力。

- Skill 设计规范；
- 产品定义、设计评审、架构评审、代码审查、用户验收等核心 Skills；
- 触发条件、不适用情况、输入输出、验证和失败升级；
- Skill 与 Context、Harness、Agent、Loop 的契约；
- Claude Code 与 Codex 第一批适配；
- 候选、已验证和稳定 Skill 的成熟度管理。

退出标准：同一类任务在至少两个执行平台上得到结构一致、可验证的产物。

## v0.4：参考工程闭环

目标：验证完整产品价值生命周期。

- 至少两类真实参考工程；
- 战略价值到产品定义的可追溯性；
- 高保真预览到代码实现的视觉验证；
- 质量、安全和模拟用户验收；
- 发布、回滚和运行反馈；
- 成本、人工修正比例和任务成功率记录；
- Framework 缺陷清单和修正。

退出标准：Framework 能够通过真实结果证明价值，而不是只形成完整文档。

## v0.5：Agent 编排与多级 Loop

目标：在保持人工关键决策的情况下连接多个角色。

- Agent 输入输出契约；
- 工具权限和责任边界；
- 失败分类、升级、停止和人工接管；
- 任务、阶段、产品和框架四级 Loop；
- 可视化状态、成本和验证报告；
- 平台能力差异和降级策略。

退出标准：多个 Agent 可以在清晰契约下协作，失败不会无限重试或越权扩张。

## v1.0：跨平台可验证框架

目标：形成稳定的标准、模板、Skills、门禁、适配和参考实现体系。

v1.0 应能够明确：

- 哪些是平台无关的核心标准；
- 哪些能力需要 Claude Code、Codex、Kimi、GLM 等平台适配；
- 哪些模板和 Skills 已经跨项目验证；
- 哪些门禁能够自动执行；
- 哪些关键节点必须保留人工责任；
- Framework 如何根据真实反馈持续演进。

v1.0 不以“完全无人化”为成功标准，而以可控、可验证、可迁移、可维护和真实产品价值为标准。
