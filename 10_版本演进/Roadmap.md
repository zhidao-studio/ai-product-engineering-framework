版本规则：[版本管理规范](版本管理规范.md)。

- 完成进入 v0.2.0 / A 真实业务验证前的清场检查。

- 专有许可证和使用边界存在唯一权威来源；
- 所有当前入口不再将项目描述为开放或开源；

- 未经授权的外部贡献和外部数据传输被明确禁止；

状态：**已通过。**

## v0.2.0：Context 与 Harness 可执行化（开发中）

目标：让一次真实 AI 任务能够获得正确 Context，在明确边界和门禁下完成，并留下可验证、可回写的证据。

### 里程碑 A：Context 可执行化（当前）

已完成：

- 框架、项目、阶段、任务四级 Context 作用域；
- 独立决策与经验回写层；
- 权威事实源、状态、版本、新鲜度和敏感级别；
- 项目 Context Pack；
- 阶段 Context；
- 任务 Context Pack；
- Context 装配、裁剪、预算和安全规则；
- 冲突、缺失和过期处理；
- 决策、证据和经验回写；
- Context 完整性检查清单；
- 第一批候选模板；
- Framework 自应用基线检查，93/100，人工确认通过；
- 统一机器状态枚举和 Pack 版本字段。

仍需完成：

- 选择经授权的真实业务参考工程；
- 完成一个低到中风险业务任务；
- 记录填写成本、人工修正、冲突、遗漏和任务结果；
- 根据业务验证修正规范和模板；
- 判断模板是否达到 `single_project_validated`。

退出标准：新的人员或 Agent 不依赖历史会话，也能在经授权的真实业务任务中恢复背景、识别权威事实、发现冲突、明确修改范围，并完成结果回写。

### 里程碑 B：Harness 可执行化

建设内容：

- 标准任务准备门禁；
- 阶段进入与退出门禁；
- 文件和模块修改边界；
- OpenAPI、Schema、依赖和行为契约门禁；
- 权限、安全、成本和停止条件；
- 三层验证模板；
- 模拟用户验收与发布门禁分离；
- 变更、发布和回滚清单；
- Markdown 链接、Mermaid、版本、状态、许可证和 Context 元数据检查；
- 第一套可运行门禁脚本。

退出标准：在参考工程中证明门禁能够发现真实问题，并且不会因为过度规则阻止正常任务。

### 里程碑 C：Context 与 Harness 联合验证

建设内容：

- 使用一个小而完整的参考任务；
- 使用项目、阶段和任务 Context Pack；
- 使用 Harness 门禁完成受控实现；
- 记录 Context 缺失、人工修正、任务成功率、成本和验证结果；
- 回写 Framework 缺陷和模板改进。

退出标准：Context 与 Harness 形成一次完整闭环，不只是文档规范。

> 里程碑 A 的业务验证与里程碑 B 可以交叉推进。先在真实任务中使用人工 Harness，识别最值得自动化的门禁，再进入里程碑 C。

## v0.3.0：Skills 标准化

目标：把已经通过真实任务验证的方法封装为能力。

- Skill 设计规范；
- 产品定义、设计评审、架构评审、代码审查、用户验收等核心 Skills；
- 触发条件、不适用情况、输入输出、验证和失败升级；
- Skill 与 Context、Harness、Agent、Loop 的契约；
- Claude Code 与 Codex 第一批适配；
- 同一任务在至少两个执行平台上的结构一致性验证。

## v0.4.0：参考工程闭环

目标：验证完整生命周期。

- 至少两类经授权的真实参考工程；
- 高保真预览到代码实现的视觉验证；
- 用户验收和运行反馈；
- 成本、人工修正比例和任务成功率记录；
- Framework 缺陷清单和修正。

## v0.5.0：Agent 编排与多级 Loop

目标：在保持人工关键决策的情况下连接多个角色。

- Agent 输入输出契约；
- 失败分类与升级；
- 任务、阶段、产品和框架四级 Loop；
- 可视化状态和验证报告。

## v1.0.0：跨平台可验证框架

目标：形成稳定的专有标准、模板、Skills、门禁、适配和参考实现体系。

v1.0.0 不以“完全无人化”为成功标准，而以可控、可验证、可迁移和真实产品价值为标准。
