## v0.2.0：Context 与 Harness 可执行化（开发中）

目标：让一次真实 AI 任务能够获得正确 Context，在明确边界和检查关卡下完成，并留下可验证、可回写的证据。

### 里程碑 A：Context 可执行化（当前）

已完成：

- 框架、项目、阶段、任务四级 Context 作用域；
- 独立决策与经验回写层；
- 权威事实源、状态、版本、新鲜度和敏感级别；
- 项目、阶段和任务 Context Pack；
- Context 装配、裁剪、预算和安全规则；
- 冲突、缺失、过期、决策、证据和经验回写；
- Context 完整性检查清单和第一批候选模板；
- Framework 自应用基线检查，93/100，人工确认通过；
- 以 YouYu“钓点详情风险摘要与风险抽屉”完成首轮真实业务参考任务的部分验证；
- 验证项目、阶段和任务 Context、修改边界、Xcode 源码与资源注册、mPaaS 真机架构构建和 Debug 路由；
- 建立 `REF-CHECK-001`，将业务验证结果、缺口和 Harness 问题回写 Framework。

仍需完成：

- 完成 YouYu 签名真机模拟用户验收；
- 关闭 P0/P1 体验问题并留存截图、录屏和设备证据；
- 记录 Context Pack 填写成本、人工修正、遗漏、冲突和任务成本；
- 根据完整业务验证修正规范和模板；
- 判断模板是否达到 `single_project_validated`；
- 由项目维护者批准里程碑 A 退出。

退出标准：新的人员或 Agent 不依赖历史会话，也能在经授权的真实业务任务中恢复背景、识别权威事实、发现冲突、明确修改范围，完成三层验证并回写结果；同时有足够证据证明模板成本与任务风险匹配。

### 里程碑 B：Harness 可执行化

建设内容：

- 标准任务准备检查关卡；
- 阶段进入与退出检查关卡；
- 文件和模块修改边界；
- OpenAPI、Schema、依赖和功能行为约定一致性检查；
- 权限、安全、成本和停止条件；
- 三层验证模板；
- 模拟用户验收与发布前检查关卡分离；
- 变更、发布和回滚清单；
- Markdown 链接、Mermaid、版本、状态、许可证和 Context 元数据检查；
- 第一套可运行检查关卡脚本；
- PR 任务状态、Required Checks、人工批准和实际合并权限的一致性控制；
- Xcode 源码注册、资源注册、真机架构和运行路由候选检查关卡。

退出标准：在参考工程中证明检查关卡能够发现真实问题，并且不会因为过度规则阻止正常任务；文档中的人工检查关卡必须能够约束实际合并行为，而不是仅作为说明存在。

### 里程碑 C：Context 与 Harness 联合验证

建设内容：

- 使用一个小而完整的前后端业务切片；
- 使用项目、阶段和任务 Context Pack；
- 使用 Harness 检查关卡完成受控实现；
- 覆盖前端、后端、OpenAPI、Schema、依赖和联调；
- 记录 Context 缺失、人工修正、任务成功率、成本和验证结果；
- 回写 Framework 缺陷和模板改进。

退出标准：Context 与 Harness 形成一次完整闭环，不只是文档规范；前端、后端、接口约定和数据结构约定能够在同一任务链路中被验证。

> 里程碑 A 的完整验收与里程碑 B 可以交叉推进。先从 YouYu 已暴露的真实问题建设最有价值的检查关卡，再在前后端完整业务切片中完成里程碑 C。

## v0.3.0：Skills 标准化

目标：把已经通过真实任务验证的方法封装为能力。

- Skill 设计规范；
- 产品定义、设计评审、架构评审、代码审查、用户验收等核心 Skills；
- 触发条件、不适用情况、输入输出、验证和失败升级；
- Skill 与 Context、Harness、Agent、Loop 的协作约定；
- Claude Code 与 Codex 第一批适配；
- 同一任务在至少两个执行平台上的结构一致性验证。

## v0.4.0：参考工程闭环

目标：验证完整生命周期。

- 至少两类经授权的真实参考工程；
- 高保真预览到代码实现的视觉验证；
- 用户验收和运行反馈；
- 成本、人工修正比例和任务成功率记录；
- Framework 缺陷清单和修正。

## v0.5.0：Agent 编排与多级 Loop

目标：在保持人工关键决策的情况下连接多个角色。

- Agent 输入输出约定；
- 失败分类与升级；
- 任务、阶段、产品和框架四级 Loop；

## v1.0.0：跨平台可验证框架

v1.0.0 不以“完全无人化”为成功标准，而以可控、可验证、可迁移和真实产品价值为标准。
